CLAUDE.md を書き始めた最初の1週間は、失敗の連続でした。振り返ると、ほとんどが「書きすぎ」か「書き方が悪い」のどちらかです。

## 失敗1: 全部書いた

「Claude Code にルールを渡せるなら、思いつく限り書いておこう」と考えました。

```markdown
- コードを書く前にテストを確認する
- エラーメッセージをよく読んでから対処する
- 変更前に git status を確認する
- 1つの関数は30行以内にする
- README を常に最新に保つ
- 外部ライブラリを追加する前に確認する
```

1週間後、これらは全部消しました。

Claude Code はこの程度のことは言われなくてもやります。わざわざ書くと、本当に重要なルール（「Chrome のプロセスを全部殺すな」のような取り返しのつかないもの）がノイズに埋もれます。

**学び**: CLAUDE.md は「一般的なベストプラクティス」を書く場所ではない。「このプロジェクト固有の文脈」を書く場所。

## 失敗2: 「〜しないこと」だけ書いた

```markdown
悪い例: bat ファイルを Write ツールで書き換えないこと
```

Claude は律儀に守ります。でも、代わりにどうすればいいか分からないので、bat ファイルに触る必要があるたびに「どうしましょうか？」と聞いてきます。

```markdown
良い例: Write ツールは常に LF で書き込むため、bat ファイルを Write で上書きすると壊れる（cmd.exe は CRLF 必須）。
bat ファイルを編集する場合は Edit ツールで部分修正するか、Write 後に sed -i 's/$/\r/' で CRLF に変換する。
```

理由と代替手段を書くと、Claude は自分で判断できます。

**学び**: 「するな」だけでなく「なぜダメで、代わりにどうするか」まで書く。

## 失敗3: 確認ポイントが曖昧だった

```markdown
外部サービスへの操作は確認してから行うこと
```

これを書いた結果、Claude は git push のたびに「push していいですか？」と聞いてきました。1日に10回以上です。

かといって「確認なしで進めてよい」と書くと、今度は本番デプロイも勝手にやります。

**学び**: 「確認する」「しない」の二択ではなく、操作ごとに具体的に決める必要がある。これは第5章で進化の実例として触れます。

## 失敗4: セッションの前提を書かなかった

CLI で使うことを前提にした回答が返ってきて混乱しました。

「ターミナルで Ctrl+C を押してプロセスを停止してください」と言われても、デスクトップアプリにはターミナルがありません。「`exit` でセッションを終了してください」と言われても、デスクトップアプリはウィンドウを閉じるだけです。

```markdown
- ユーザーは Claude Code デスクトップアプリ（Windows）を主に使っている
- セッション終了は「ウィンドウを閉じる」。exit や Ctrl+C の案内は不適切
```

この2行を追加しただけで、的外れな案内がなくなりました。

**学び**: 自分の環境の前提は、最初に書く。Claude は聞かないと分からない。

## この章のまとめ

最初の1週間の失敗は、すべてこの1文に集約されます。

> CLAUDE.md は「Claude に知っていてほしいこと」を書く場所ではなく、「書かないと事故が起きること」を書く場所。

次の章から、1ヶ月の運用を経て「残ったルール」「消えたルール」「変わったルール」を具体的に見ていきます。
