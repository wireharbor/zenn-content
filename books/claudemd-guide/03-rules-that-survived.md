1ヶ月後も生き残っているルールには共通点があります。すべて、実際にやらかした失敗から生まれたものです。

ここでは「どんな種類のルールが残るか」のパターンを紹介します。

## パターン1: 取り返しがつかない事故の防止

あるプロセスを全部殺したら、ユーザーが作業中だった別のアプリケーションも巻き添えになった。あるファイルを「参照なし」と判断して消したら、外部の自動化タスクが壊れた。

こういう「1回やったら取り返しがつかない事故」から生まれたルールは消えません。消すと同じ事故が起きるからです。

具体例を1つ見せます。これは実際に1ヶ月残っているルールです（匿名化済み）。

```markdown
ファイルの移動・削除前に全参照元を確認する。
以下を全て grep すること:
1. ワークスペース内のコード
2. 自動化設定ディレクトリ（~/.config/, ~/.local/ 等）
3. スケジュールタスクの定義ファイル
4. その他の設定ファイル・環境設定

理由: 2回連続で、ワークスペース内の grep だけで「参照なし」と判断して
ファイルを消し、外部の自動化タスクが壊れた。チェック対象を省略するたびに
壊れるので、省略しない仕組みにした。
```

注目してほしいのは3つの特徴です。

1. **操作が具体的**: 「安全に配慮する」ではなく「grep する対象リスト」が列挙されている
2. **理由が具体的**: 「2回連続で壊した」という事故回数と結果が書いてある
3. **チェックリスト形式**: Claude がそのまま手順として実行できる

こういうルールは「全文を載せなくてもパターンだけで」とは言えません。**粒度そのものが生存の条件**です。残るルールの全文と各セクションの判断理由は、第11章でまとめて掲載しています。

## パターン2: 同じ失敗の繰り返し防止

同じエラーに対して「待ち時間を伸ばす」という修正を3回繰り返して、3回とも失敗した。根本原因は待ち時間ではなく、API の使い方だった。

Claude は指示されないと、同じ方向の修正を何度でも繰り返します。「3回同じ方向の修正をしたら立ち止まれ」と書くと、このループが止まります。

## パターン3: 判断基準の明示

Claude は「おそらくこの API があるはずです」と推測で修正を始めることがあります。「推測で動くな、調べてから動け」と書いておくと、推測の頻度が大幅に減ります。

このルールはあるプロジェクトで書いた後、どのプロジェクトでも同じ問題が起きることに気づいて、グローバルルールに昇格させました。

## 残るルールの共通点

残るルールに共通するのは4つの特徴です。

1. **事故起点**: 実際に起きた事故から書かれた
2. **具体的**: 抽象論ではなく特定の操作に紐づいている
3. **理由付き**: なぜそのルールが必要かが書いてある
4. **消すと壊れる**: ルールを外すと同じ事故が再発する

逆に言えば、**「何も起きていないのに予防的に書いたルール」は残らない**ということです。これが次の章のテーマです。
