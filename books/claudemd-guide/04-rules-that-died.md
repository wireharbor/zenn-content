最初の1週間で書いて、1ヶ月以内に消したルールがたくさんあります。消した理由にはパターンがあります。

消したルールの具体的な一覧は、有料パートの第11章に掲載しています。ここではパターンだけ紹介します。

## パターン1: 当たり前すぎるもの

Claude Code は基本的なソフトウェアエンジニアリングのプラクティスを知っています。「テストを確認してから実装する」「エラーメッセージを読む」のようなルールは、書かなくても Claude はやります。

わざわざ書くと、本当に重要なルールがノイズに埋もれます。

**判断基準**: 「このルールを書かなかったら、Claude は違う行動をするか？」。答えが No なら不要。

## パターン2: 過剰な制約

「1つの関数は30行以内」のようなルールは、Claude の判断を不必要に縛ります。Claude は文脈に応じて適切な判断ができるのに、一律の数値制約を設けると、無理な分割で可読性が下がることがあります。

**判断基準**: 「このルールがないと、Claude はひどいコードを書くか？」。答えが No なら、Claude の判断に任せたほうがいい。

## パターン3: 文脈に合わないもの

一般的には正しいが、このプロジェクトの現実に合わないルール。たとえば日本語主体のプロジェクトで「コミットメッセージは英語で」と書くと、他の部分でも不自然な判断をしがちです。

**判断基準**: 「このルールは、このプロジェクトの現実に合っているか？」

## パターン4: 運用コストが高すぎるもの

「README を常に最新に保つ」は正しく見えますが、Claude は小さな変更のたびに README を書き換えようとします。README の diff が大量に発生して、本来のコード変更が見えなくなります。

**判断基準**: 「このルールを守るコストは、守らないリスクより小さいか？」

## 消すタイミング

ルールを消すタイミングには2つのパターンがあります。

**即消し**: 書いた翌日に「これは不要だ」と気づく。パターン1がこれに当たります。

**じわじわ消し**: 2週間ほど運用して、ルールが一度も発動しなかったことに気づく。パターン2〜4がこれに当たります。

具体的な消す基準と棚卸しの方法は、第10章「効くルールの書き方」のテクニック7で詳しく扱います。

## この章のまとめ

> CLAUDE.md のルールは「足りないから追加する」より「多すぎるから削る」方が難しい。
> 消す基準を持たないと、ファイルは肥大化する一方。
