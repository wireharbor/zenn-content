「残った」でも「消えた」でもない第3のカテゴリがあります。**内容が変わった**ルールです。

共通点は、最初は抽象的だったものが、運用するうちに具体的な条件文に進化していったことです。

## 実例1: 外部操作の確認ルール

最初は「外部サービスへの操作は確認してから行うこと」の1行でした。

1週間後には「push は確認、デプロイは確認、ファイル操作は確認なし」の3行になりました。

1ヶ月後には、操作ごとに「確認なし / 確認必要 / 禁止」を細かく分けた8行以上のリストに進化しています。具体的なリストは第11章で全文を掲載しています。

**なぜ変わったか**: 最初の「全部確認」だと、git push のたびに止まって作業が進みません。「全部自由」にすると本番事故が起きます。1ヶ月かけて、操作ごとの「本当のリスク」が見えてきた結果、この粒度に落ち着きました。

## 実例2: ブラウザ操作の使い分け

最初は「ブラウザ操作は headless で行う」の1行でした。

1ヶ月後には「どの操作が headless で動き、どの操作に headed が必要か」の判断表と、「API で代替できるなら API を優先する」という原則に進化しています。

**なぜ変わったか**: 一部の Web サービスは自動操作のクリックをブロックします。「全部同じ方式で動く」は通用しませんでした。でも全部重い方式にすると無駄が多い。最終的に API で代替できるものは API に移行し、このルールは「API がないとき」の判断基準になりました。

## 実例3: 基本動作の自由度

最初は「何をするにも確認してください」の1行でした。

1ヶ月後には「ローカル操作は自由、外部送信系だけ確認」という原則に、具体的な例外リスト付きで進化しています。

**なぜ変わったか**: Claude は指示通りに「全部確認」を実行しました。1行のファイル編集にも確認を求めてきて、作業が止まります。「ここまでは自由、ここからは止まれ」の境界を明示したら、作業速度が劇的に上がりました。

## ルールが進化するメカニズム

1. **抽象ルールを書く** → 「安全に配慮する」
2. **Claude が判断に迷う** → 「これは確認が必要ですか？」と毎回聞いてくる
3. **具体的な回答をする** → 「git push は OK、PR 作成は確認して」
4. **回答をルールに書き戻す** → 条件文として定着

このサイクルを繰り返すと、ルールは自然に「抽象 → 具体」へ進化します。

## この章のまとめ

> 抽象ルールは「Claude が毎回聞いてくる」という形でフィードバックが来る。
> そのフィードバックを条件文としてルールに書き戻すと、進化する。
> 最初から完璧な条件文は書けない。運用しながら育てるしかない。
