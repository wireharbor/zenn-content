---
title: "グローバルルールとプロジェクトルールの分け方"
---

Claude Code には CLAUDE.md を置ける場所が2つあります。

- **グローバル**: `~/.claude/CLAUDE.md` — どのプロジェクトを開いても適用される
- **プロジェクト**: リポジトリのルートにある `CLAUDE.md` — そのプロジェクトでだけ適用される

最初は「全部プロジェクトに書けばいいのでは？」と思っていました。実際に複数プロジェクトを運用してみると、この2層の分離が必要な理由が分かります。

## なぜ分けるのか

### 理由1: 同じルールを何度も書く羽目になる

「推測で修正を始めるな、公式ドキュメントを確認しろ」というルールは、どのプロジェクトでも必要です。プロジェクトごとに書いていると、5つのリポジトリに同じ文が5回コピーされます。1箇所を直しても他の4箇所は古いまま残ります。

グローバルに1回書けば、全プロジェクトで自動的に適用されます。

### 理由2: プロジェクト固有の情報がグローバルに混ざると害がある

「Headless Chrome は port 9223 で起動する」というルールを、Chrome を使わないプロジェクトのグローバルに書いておくのは無意味です。無意味なだけでなく、Claude のコンテキストウィンドウを消費します。

CLAUDE.md が長くなるほど、Claude が各ルールに割ける注意力は下がります。不要なルールは、必要なルールの効果を薄めます。

### 理由3: チームで共有するとき困る

プロジェクトの `CLAUDE.md` は Git でチームに共有できます。ここに「回答は日本語で」「コミットメッセージの末尾に署名を入れて」のような個人の好みを書くと、チームメンバー全員に強制されます。

個人の好みはグローバルに、チームの規約はプロジェクトに、が正しい分離です。

## 何をグローバルに書くか

グローバルに置くべきルールは3種類です。

### 種類1: 実行環境の前提

```markdown
- ユーザーは Claude Code デスクトップアプリ（Windows）を主に使っている
- セッション終了は「ウィンドウを閉じる」。exit や Ctrl+C の案内は不適切
```

OS やエディタの情報は、どのプロジェクトでも変わりません。ここに書いておくと、Claude が毎回 CLI 前提の回答をする問題がなくなります。

### 種類2: Claude の振る舞いに対するガードレール

```markdown
- 「〜だと思う」で修正を始める前に、公式ドキュメントを確認する
- 同じ不具合が3回再発したら、別のアプローチを検討する
```

これらは Claude の「癖」に対するルールです。推測で動く、同じ修正を繰り返す、といった振る舞いはプロジェクトに依存しません。

### 種類3: 出力フォーマットの好み

```markdown
- Web URL は markdown リンクにする
- ローカルファイルパスはコードブロックで出す
- 回答は日本語で
```

見た目や言語の好みは完全に個人的なものなので、グローバルに書きます。

## 何をプロジェクトに書くか

プロジェクトに置くべきルールは4種類です。

### 種類1: ツールの接続先・ポート番号

```markdown
- Headless Chrome は port 9223 で起動する
- API サーバーは localhost:3000
- データベースは Docker の PostgreSQL、ポート 5432
```

### 種類2: 外部操作の分類

操作ごとに「確認なし / 確認必要 / 禁止」を分ける分類です。同じ操作でも、プロジェクトによってリスクが違います。個人プロジェクトなら push は自由、チームプロジェクトなら確認必要、ということがあります。

具体的な分類の作り方と実例は、第12章のパターン1で詳しく扱います。

### 種類3: 文書の優先順位

```markdown
- 正本は WORKSPACE_RULES_INDEX.md
- 文書の優先順位: この CLAUDE.md > 各サブディレクトリの README
```

### 種類4: プロジェクト固有の技術的制約

```markdown
- bat ファイルの改行は CRLF 必須。Write ツールは LF で書くので注意
- shared core ディレクトリは複数人が同時に編集しない
```

## グローバルに「昇格」させるタイミング

最初はプロジェクト固有として書いたルールが、後からグローバルに昇格することがあります。

### 昇格の判断基準

この3つを全て満たしたらグローバルに昇格させます。

1. **汎用的**: 特定プロジェクトに依存しない
2. **再発性**: 他のプロジェクトでも同じ問題が起きる可能性がある
3. **実績**: 2つ以上のプロジェクトで適用した実績がある

### 実際に昇格させた例

| ルール | 最初の場所 | 昇格の理由 |
|--------|-----------|-----------|
| 調査してから動く | プロジェクト | 全プロジェクトで推測による誤修正が発生した |
| 同じ不具合は別アプローチで | プロジェクト | Claude の振る舞いに起因する問題で、プロジェクト非依存 |
| 全プラットフォーム同一品質 | プロジェクト | 複数プラットフォーム対応は特定プロジェクトの話ではなかった |

### 昇格の手順

1. グローバルの `~/.claude/CLAUDE.md` にルールを追加する
2. プロジェクト側からは同じルールを削除する
3. プロジェクト側に残すのは、グローバルルールへの**例外**だけ

例えば、グローバルに「push は確認が必要」と書いてあるが、このプロジェクトでは「push は確認なしでOK」にしたい場合、プロジェクト側に例外として書きます。

## 迷ったときの判断フロー

```
このルール、他のプロジェクトに持っていっても意味がある？
  ├─ Yes → グローバル候補
  │   └─ 2つ以上のプロジェクトで同じ問題が起きた？
  │       ├─ Yes → グローバルに昇格
  │       └─ No → まだプロジェクトに置く。2つ目で昇格
  └─ No → プロジェクトに置く
```

## この章のまとめ

- グローバルには「環境の前提」「Claude の癖への対策」「出力の好み」を書く
- プロジェクトには「接続先」「操作の分類」「技術的制約」を書く
- 同じルールを2つ以上のプロジェクトに書いたら、グローバルに昇格させる
- 迷ったら「他のプロジェクトでも意味があるか」で判断する
