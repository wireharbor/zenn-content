前の章では「グローバルとプロジェクト」の2層を扱いました。実は Claude Code のルール管理はもっと細かく、4つの層があります。

この章では4層の全体像と使い分けを解説します。これを理解すると、CLAUDE.md に何を書き、何を別の場所に置くべきかが明確になります。

## 4層の全体像

| 層 | ファイル | 役割 | 共有範囲 |
|---|---|---|---|
| 1. グローバルルール | `~/.claude/CLAUDE.md` | 全プロジェクト共通の行動指針 | ローカルのみ |
| 2. プロジェクトルール | `<repo>/CLAUDE.md` | リポジトリ固有の文脈 | Git で共有 |
| 3. パスベースルール | `<repo>/.claude/rules/*.md` | ファイルパターンに紐づくルール | Git で共有 |
| 4. Memory | `~/.claude/projects/*/memory/` | 対話から学んだ知識 | ローカルのみ |

Claude がセッションを開始すると、この4層が上から順に読み込まれます。下の層のルールは上の層のルールを上書きできます（プロジェクトルールがグローバルルールの例外を定義する、など）。

## 層1: グローバルルール

前章で詳しく扱ったので、ここではポイントだけ。

- **置き場所**: `~/.claude/CLAUDE.md`
- **適用範囲**: Claude Code で開くすべてのプロジェクト
- **典型的な内容**: OS・エディタの前提、Claude の癖への対策、出力フォーマットの好み
- **注意点**: ここに書いたものは全プロジェクトに影響する。プロジェクト固有の情報を書くとノイズになる

## 層2: プロジェクトルール

これも前章で扱いました。補足として：

- **置き場所**: リポジトリのルートにある `CLAUDE.md`
- **適用範囲**: そのリポジトリで作業するとき
- **Git で共有できる**: チームメンバーが同じルールで作業できる
- **典型的な内容**: 技術スタック、接続先、操作の許可レベル、コード規約

## 層3: パスベースルール（.claude/rules/）

ここからが本章の新しいトピックです。

### パスベースルールとは

`.claude/rules/` ディレクトリに Markdown ファイルを置くと、ファイルの frontmatter で指定した条件に合うファイルを編集するときだけ、そのルールが適用されます。

```
.claude/
└── rules/
    ├── api-guidelines.md
    ├── test-rules.md
    └── frontend-style.md
```

### なぜ必要か

CLAUDE.md にすべてを書くと、ファイルが肥大化します。テストの規約、API の設計規約、フロントエンドのスタイル規約......全部を1つのファイルに入れると200行、300行とすぐに膨れます。

パスベースルールを使うと、「テストファイルを触るときだけテスト規約を読む」「API ディレクトリのファイルを触るときだけ API 規約を読む」という形で、関心の分離ができます。

### 書き方の例

```markdown
---
description: テストファイルの規約
globs: ["**/*.test.ts", "**/*.spec.ts"]
---

- テストには AAA パターン（Arrange-Act-Assert）を使う
- モックは最小限にし、実際のデータベースを使う統合テストを優先する
- テスト名は「〜するとき、〜になる」の形式で書く
```

```markdown
---
description: API エンドポイントの規約
globs: ["src/api/**/*.ts"]
---

- エンドポイントごとに入力バリデーションを行う
- エラーレスポンスは RFC 7807 (Problem Details) に従う
- 新しいエンドポイントには OpenAPI 定義を先に書く
```

### CLAUDE.md との使い分け

| 書く場所 | 使うとき |
|---------|---------|
| CLAUDE.md | プロジェクト全体に適用するルール |
| .claude/rules/ | 特定のファイルパターンやディレクトリにだけ適用するルール |

**判断基準**: 「このルール、全ファイルに効いてほしい？それとも特定の種類のファイルだけ？」

### 注意点

- ルールファイルが多すぎると管理が大変。5〜10ファイル程度が実用的な上限
- ルール間で矛盾があると Claude が混乱する。矛盾する可能性があるルールは、同じファイルに書いて優先順位を明示する
- glob パターンの書き方を間違えると、意図しないファイルにルールが適用される

## 層4: Memory

Memory は他の3層とは性質が異なります。ルール（「こうしろ」）ではなく、知識（「〜を学んだ」）を保存する仕組みです。

Memory には user、feedback、project、reference の4タイプがあり、それぞれ異なる種類の知識を保存します。4タイプの詳しい説明と活用法は次の章で扱います。

ここで押さえておくべきポイントは、**CLAUDE.md と Memory を混同しない**ことです。CLAUDE.md はルール（命令）、Memory は知識（記録）。「認証ミドルウェアのリファクタリングが進行中」のような進行状況を CLAUDE.md に書くと、リファクタリングが終わっても消し忘れて、Claude が古い制約を守り続けます。進行状況は Memory に書きましょう。

## 4層の使い分けまとめ

```
「全プロジェクトで常に適用」
    → 層1: グローバルルール (~/.claude/CLAUDE.md)

「このプロジェクトで常に適用」
    → 層2: プロジェクトルール (CLAUDE.md)

「特定ファイルを触るときだけ適用」
    → 層3: パスベースルール (.claude/rules/)

「Claude が対話から学んだ知識」
    → 層4: Memory (.claude/projects/*/memory/)
```

迷ったら、まず層2（プロジェクト CLAUDE.md）に書きます。それで十分です。ファイルが肥大化してきたら層3に分離し、他のプロジェクトでも使うなら層1に昇格させます。

## この章のまとめ

- Claude Code のルール管理は4層構造
- CLAUDE.md（グローバル/プロジェクト）は「命令」、rules/ は「条件付き命令」、Memory は「知識」
- 最初は CLAUDE.md だけで十分。肥大化してきたら他の層に分離する
- Memory と CLAUDE.md を混同しない。進行状況は Memory、ルールは CLAUDE.md
