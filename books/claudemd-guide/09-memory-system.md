title: "Memory システム — Claude が覚えていてくれる仕組み"
---

CLAUDE.md が「ルールブック」なら、Memory は「業務日誌」です。

Claude Code は会話の中で学んだことを Memory に保存し、次のセッションで参照します。CLAUDE.md は手動で書きますが、Memory は Claude が自分で書きます。

## Memory の仕組み

### ファイル構成

Memory は2つの要素で構成されています。

1. **MEMORY.md**: インデックス（目次）。Claude がセッション開始時に読む
2. **個別ファイル**: 各トピックの詳細。MEMORY.md からリンクされている

```
~/.claude/projects/<project-hash>/memory/
├── MEMORY.md              ← インデックス（目次）
├── user_role.md            ← ユーザー情報
├── feedback_testing.md     ← フィードバック
├── project_auth_refactor.md ← プロジェクト状況
└── reference_linear.md     ← 外部参照
```

### MEMORY.md の例

```markdown
- [ユーザーの役割](user_role.md) — データサイエンティスト、フロントエンド経験は浅い
- [テスト方針](feedback_testing.md) — モックではなく実DBを使う
- [認証リファクタ進行中](project_auth_refactor.md) — コンプライアンス要件で書き直し中
- [バグ管理](reference_linear.md) — Linear の INGEST プロジェクト
```

Claude はセッション開始時にこのインデックスを読み、関連しそうなトピックの詳細ファイルを必要に応じて参照します。

### 個別ファイルの書き方

各ファイルには frontmatter（メタデータ）と本文があります。

```markdown
---
name: テスト方針の修正
description: ユニットテストでモックを使わず実DBを使う方針に修正された
type: feedback
---

テストでデータベースのモックを使ったら、モックと本番の挙動が違ってバグを見逃した。
以降、統合テストでは実DBを使う。

**Why:** 前四半期にモックテストが通ったのに本番マイグレーションが壊れた事故
**How to apply:** テストを書くとき、DBアクセスがあるテストはモックではなく実DBに接続する
```

## Memory の4タイプ

:::message
Memory のタイプ分類（user / feedback / project / reference）は、筆者が自分の運用で定義したものです。Claude Code の公式機能として4タイプが組み込まれているわけではありません。Memory ファイルの frontmatter に `type` フィールドを設けて分類する運用を筆者が行っており、Claude はそれに従って動作しています。タイプの名前や分類基準は、あなたのプロジェクトに合わせて自由に変えて構いません。
:::

### 1. user: ユーザー情報

Claude がユーザーに合わせた応答をするための情報。

**いつ保存されるか**: ユーザーが自分の役割や経験を話したとき。

```
「私はバックエンドエンジニアで、フロントエンドは初めてです」
→ Claude がフロントエンドの説明をバックエンドの概念に例えて説明するようになる
```

### 2. feedback: 行動フィードバック

「こうしろ」「こうするな」という修正指示。CLAUDE.md に書くほどではないが、覚えておいてほしいこと。

**いつ保存されるか**: ユーザーが Claude の行動を修正したとき。

```
「その書き方じゃなくて、このプロジェクトでは arrow function を使って」
→ 次回から arrow function で書くようになる
```

**CLAUDE.md との違い**: feedback は「このセッションで学んだこと」で、他のプロジェクトに持ち運ぶ必要がないもの。持ち運ぶべき汎用ルールは CLAUDE.md に書きます。

### 3. project: プロジェクト状況

進行中のタスクや決定事項。コードや Git から読み取れない「文脈」。

**いつ保存されるか**: プロジェクトの背景や決定理由が話題に出たとき。

```
「この認証システムを書き直しているのは、法務からセッションの保存方法にNGが出たから」
→ リファクタのスコープ判断で、技術的な最適化よりコンプライアンスを優先するようになる
```

### 4. reference: 外部参照

情報がどこにあるかのポインタ。

**いつ保存されるか**: 外部システムの場所を教えたとき。

```
「バグの管理は Linear の INGEST プロジェクトでやっている」
→ バグ関連の質問をされたとき「Linear の INGEST を確認しましょう」と提案するようになる
```

## Memory を効果的に使うコツ

### コツ1: 明示的に「覚えておいて」と言う

Claude は会話の中から自動的に Memory を作成しますが、確実に覚えてほしいことは「これを Memory に保存して」と明示的に言う方が確実です。

```
「このプロジェクトでは、テストは pytest ではなく pytest-asyncio を使う。覚えておいて」
```

### コツ2: 定期的に棚卸しする

Memory は増える一方です。古い project メモリは特に劣化が早くなります。「あのリファクタリング進行中」という Memory が、リファクタ完了後も残っていると、Claude が古い情報をもとに判断します。

月に1回、MEMORY.md を眺めて古い項目を消す習慣をつけると、Memory の精度が保たれます。

### コツ3: CLAUDE.md と Memory を区別する

| こう思ったら | 書く場所 |
|------------|---------|
| 「毎回このルールを守ってほしい」 | CLAUDE.md |
| 「この文脈を覚えておいてほしい」 | Memory |
| 「今回だけこうしてほしい」 | セッション内の指示（保存しない） |

### コツ4: Memory から CLAUDE.md に昇格させる

feedback タイプの Memory が何度も参照されていたら、それは CLAUDE.md に書くべきルールです。

```
Memory: 「テストはモックではなく実DBを使う」（feedback）
→ 3回以上参照された
→ CLAUDE.md に昇格: 「テストでは実DBを使う統合テストを優先する」
```

## Memory に保存しないもの

以下は Memory に保存しても意味がないか、害があるものです。

- **コードの構造**: `git log` や `grep` で分かる
- **ファイルパス**: 次に見たとき移動しているかもしれない
- **デバッグの詳細**: 修正はコードに反映済み、経緯はコミットメッセージにある
- **一時的な作業状態**: 「今 branch-x で作業中」はセッション内で十分

## この章のまとめ

- Memory は Claude が対話から学んだ知識を保存する仕組み
- 4タイプ: user（ユーザー情報）、feedback（修正指示）、project（進行状況）、reference（外部参照）
- CLAUDE.md はルール（命令）、Memory は知識（記録）。混同しない
- 定期的に棚卸しして古い Memory を消す
- 何度も参照される feedback は CLAUDE.md に昇格させる
