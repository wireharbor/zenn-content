---
title: "効くルールの書き方 — 7つのテクニック"
---

CLAUDE.md に何を書くかは前の章で扱いました。この章では「どう書くか」を扱います。

同じルールでも、書き方によって Claude への効き方が大きく変わります。1ヶ月の運用で見つけた7つのテクニックを紹介します。

## テクニック1: 理由を書く

「〜しないこと」だけだと、Claude は例外ケースで判断できません。

```markdown
❌ main ブランチに直接 push しないこと
```

Claude はこのルールを律儀に守ります。でも、feature ブランチにいるときにも「push していいですか？」と聞いてきます。どのブランチが OK なのか分からないからです。

```markdown
✅ main ブランチへの直接 push は禁止（レビューなしの変更が本番に入るため）。
   feature ブランチへの push は自由。
   main への変更は PR 経由で行う。
```

理由が書いてあると、Claude は「今いるのは feature ブランチだから push して大丈夫」と自分で判断できます。想定外のブランチにいた場合も「これは main に近い性質のブランチだから確認しよう」と推論できます。

**原則**: ルールの後に必ず「なぜなら〜」を書く。

## テクニック2: Good / Bad の例を入れる

抽象的な指示より、具体的な例が効きます。

```markdown
❌ コミットメッセージは分かりやすく書くこと
```

```markdown
✅ コミットメッセージの例:
   Good: "fix: ログイン時にセッションが二重作成される問題を修正"
   Bad: "バグ修正"
   Bad: "色々直した"
```

Claude は大量のコードを学習しているので、例を見ると「あ、このスタイルか」と即座に理解します。言葉で説明するよりも例の方が確実に伝わります。

**原則**: 迷いそうな判断には Good / Bad の対を入れる。

## テクニック3: 「するな」には代替手段を添える

禁止だけだと Claude は止まります。

```markdown
❌ git add -A は使わないこと
```

```markdown
✅ git add -A は使わない（他のスレッドの staged changes を巻き込む可能性がある）。
   代わりに、変更したファイルを個別に git add する。
```

禁止 + 理由 + 代替手段の3点セットが効果的です。

**原則**: 「するな」と書いたら、次の行に「代わりに〜する」を書く。

## テクニック4: 操作ごとに具体的な条件を書く

```markdown
❌ 安全に配慮して操作すること
```

Claude は「安全とは何か」を判断できません。結果、全操作で確認を求めてくるか、全操作を自由にやるかの両極端になります。

```markdown
✅ ファイル削除の条件:
   - テスト用の一時ファイル → 確認なしで削除してよい
   - ソースコード → 参照元を全て確認してから削除
   - 設定ファイル → 確認が必要
   - .env や認証情報 → 禁止（手動で行う）
```

操作の一覧を作って、それぞれに「確認なし / 確認必要 / 禁止」を割り当てるのが最も効果的です。外部操作全体の分類例は第12章のパターン1で詳しく扱います。

**原則**: 抽象ルールは操作ごとの条件文に分解する。

## テクニック5: 適用条件を限定する

ルールが常に適用されるとノイズになります。

```markdown
❌ 外部 API を呼ぶ前に、必ずレート制限を確認すること
```

ローカルの開発用 API にレート制限はありません。常に適用するルールにすると、ローカルテストのたびに確認が入ります。

```markdown
✅ 本番環境の外部 API を呼ぶ前に、レート制限を確認する。
   ローカル開発用の API（localhost）は対象外。
```

**原則**: 「いつ適用するか」「いつ適用しないか」を明示する。

## テクニック6: 階層で粒度を変える

全部を同じ粒度で書くと、重要なルールが埋もれます。

```markdown
❌ すべてのルールを箇条書きで並列に並べる（30項目）
```

```markdown
✅ ## 基本動作（最重要・毎セッション適用）
   - ローカル操作は確認なし
   - 外部操作は確認が必要

   ## ファイル操作の注意点（ファイル操作時のみ参照）
   - 削除前に参照元を確認
   - bat ファイルは CRLF に注意

   ## ブラウザ操作（ブラウザ使用時のみ参照）
   - Chrome の全プロセス kill は禁止
   - headless / headed の使い分けは...
```

見出しで分けると、Claude は「今の作業に関係するセクション」に注目できます。

**原則**: 重要度と関連度でセクションを分ける。全部を同じレベルで並べない。

## テクニック7: 定期的に削る

ルールを追加し続けると、ファイルは肥大化します。肥大化したルールファイルは、ルールがないのと同じくらい効果がありません。

**削除の基準**:

1. **Claude が言われなくてもやること**: 「テストを確認する」「エラーメッセージを読む」→ 消す
2. **2週間発動しなかったルール**: ルールが発動する状況が起きていない → 本当に必要か見直す
3. **同じことを2箇所に書いている**: グローバルとプロジェクトの重複 → 1箇所に統合
4. **Claude の判断の方が適切なケース**: 「関数は30行以内」のような一律制約 → 消す

**原則**: 追加したら、同じ回数だけ削除のチャンスを作る。月1回の棚卸しが目安。

## 7つのテクニックの早見表

| # | テクニック | 一言 |
|---|----------|------|
| 1 | 理由を書く | 「なぜなら〜」を必ず添える |
| 2 | Good / Bad の例 | 抽象指示より具体例が効く |
| 3 | 代替手段を添える | 「するな」の次に「代わりに〜」 |
| 4 | 操作ごとの条件 | 抽象ルールを条件文に分解する |
| 5 | 適用条件の限定 | いつ適用し、いつ適用しないか |
| 6 | 階層で粒度を変える | 重要度と関連度でセクション分け |
| 7 | 定期的に削る | 月1回の棚卸しで不要ルールを消す |

## やってはいけない書き方

7つのテクニックの裏返しとして、2つの典型的な失敗パターンを紹介します。

### 禁止リストだけの CLAUDE.md

「〜するな」が10行並んでいるが、理由が1つも書いていない。こういう CLAUDE.md を書くと、Claude は律儀に禁止を守る一方で、例外ケースのたびに質問してきます。

```markdown
❌ こう書いた:
   - git add -A を使うな
   - rm -rf を使うな
   - main に push するな

→ ファイルを1つ追加したいだけなのに
  「git add はルール違反ですか？」と聞いてくる
```

理由（なぜ禁止か）と代替手段（代わりに何をするか）がないと、Claude は禁止の範囲を自分で判断できません。テクニック1と3を合わせて使ってください。

### 進行状況メモの混入

CLAUDE.md に「現在リファクタリング中」「API v2 に移行予定」のような進行状況を書いてしまうパターンです。

```markdown
❌ CLAUDE.md に書いた:
   - 現在 auth モジュールをリファクタリング中。旧コードは触らない。

→ リファクタリングが終わっても記述は残り、
  Claude は完了した作業の制約を守り続ける
```

進行状況は Memory に書いてください。Memory は会話や時期と紐づくので自然に古くなりますが、CLAUDE.md のルールは明示的に消すまで永久に効き続けます。ルール（恒久的な制約）と記録（一時的な状態）を混ぜないのが鉄則です。

## この章のまとめ

「何を書くか」と「どう書くか」は同じくらい重要です。

同じルールでも、理由・例・代替手段・条件を書くかどうかで、Claude の理解度と遵守率が大きく変わります。そして、書いたルールを定期的に削ることで、残ったルールの効果が維持されます。
