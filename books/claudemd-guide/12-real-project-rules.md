title: "実物公開: プロジェクトルールのパターン集"

前章ではグローバルルールの全文を掲載しました。この章では、プロジェクト固有のルール（リポジトリのルートに置く `CLAUDE.md`）のパターンを紹介します。

プロジェクトルールは内容が環境に依存するので、全文を載せても参考になりにくい部分があります。代わりに、「型」として再利用できる5つのパターンを、実例付きで紹介します。

## パターン1: 外部操作の tier 分類

操作ごとに「確認なしでOK」「確認が必要」「禁止」を分ける型です。

### 実例

```markdown
## 外部操作の分類

### 確認なしで進めてよい（Tier 1）
- git commit, push（feature ブランチ）
- ローカルの依存パッケージ追加・更新
- テスト・lint の実行
- 開発用サーバーの起動・停止
- Render の通常再デプロイ

### 確認が必要（Tier 2）
- PR の作成・マージ
- 新しい API エンドポイントの公開
- データベーススキーマの変更
- CI/CD パイプラインの変更

### 確認 + 慎重な実行（Tier 3）
- 本番環境へのデプロイ
- 本番データベースのマイグレーション
- 認証情報・シークレットの変更
- billing / plan の変更
- サービスの削除
```

### このパターンの効果

最初に「安全に操作してください」と書いていたときは、Claude が git push のたびに確認を求めてきて、1日10回以上の中断が発生しました。

tier 分類に変えてからは、中断が1日1〜2回（PR 作成やデプロイのとき）に減りました。

### 応用のコツ

- **Tier 1 を広めに取る**: ローカルで完結する操作は基本的に Tier 1。迷ったら Tier 1 に入れて、問題が起きたら Tier 2 に上げる
- **操作名を具体的に書く**: 「外部への書き込み」ではなく「PR 作成」「デプロイ」と操作名で書く
- **プロジェクトごとに違う**: 個人プロジェクトなら push は Tier 1、チームプロジェクトなら Tier 2

## パターン2: ツールの接続情報

Claude が毎回調べなくていいように、プロジェクトで使うツールの接続先を書いておく型です。

### 実例

```markdown
## ツールと接続先

- API サーバー: localhost:3000（Express）
- フロントエンド: localhost:5173（Vite）
- データベース: PostgreSQL on localhost:5432（Docker）
- Redis: localhost:6379（Docker）
- Headless Chrome: CDP on port 9223
- CI: GitHub Actions（.github/workflows/）
- デプロイ先: Render（自動デプロイ、main ブランチ）
```

### このパターンの効果

接続情報を書かないと、Claude は毎回 `docker ps` や設定ファイルを読んでポート番号を調べます。10秒で済むことですが、1セッションで5〜10回発生すると、累積で待ち時間が気になります。

### 応用のコツ

- **ポート番号は特に重要**: デフォルトと違うポートを使っている場合、Claude は間違える確率が高い
- **Docker か直接起動かを明記**: データベースが Docker なのかローカルインストールなのかで、接続方法が変わる
- **認証情報は書かない**: パスワードやトークンは `.env` に置き、CLAUDE.md には「`.env` を参照」とだけ書く

## パターン3: 文書の優先順位

複数の設定ファイルやドキュメントがあるとき、どれが正（正本）かを明示する型です。

### 実例

```markdown
## 文書の優先順位

- この CLAUDE.md が最上位の設定。他のドキュメントと矛盾する場合はこちらに従う
- API の仕様: OpenAPI 定義（api/openapi.yaml）が正本。README の記述が古い場合がある
- デプロイ手順: DEPLOYMENT.md が正本。Wiki の手順は更新が遅れている可能性あり
- 環境変数: .env.example が定義一覧。実際の値は .env（Git 管理外）
```

### このパターンの効果

プロジェクトが育つと、同じ情報が複数の場所に書かれます。README、Wiki、CLAUDE.md、コード内コメント......Claude はどれを信じるか迷います。

「正本はこれ」と明示しておくと、Claude は矛盾する情報に出会ったとき正本を優先します。

## パターン4: 並行作業のルール

サブエージェントや複数セッションで作業するときの制約を書く型です。

### 実例

```markdown
## 並行作業のルール

- 他のセッションが編集中の可能性があるファイルを上書きしない
  → git status で uncommitted changes を確認してから編集する
- git add -A や git reset --hard を使わない
  → 他のセッションの staged changes を巻き込む
- shared/ ディレクトリのファイルは並列編集しない
  → 共通の型定義やユーティリティは1人ずつ触る
- MCP サーバーの再起動は他のセッションの接続を切る
  → 再起動が必要なときは影響を報告する
```

### このパターンの効果

1人で作業しているときは気にならないルールですが、サブエージェントを使い始めると必須になります。2つのサブエージェントが同じファイルを同時に編集して、片方の変更が消える事故を防ぎます。

### 応用のコツ

- **並列化しないファイルを明示**: 「shared/ は1人ずつ」のように、並列禁止のディレクトリを書く
- **git の危険コマンドを列挙**: `git add -A`、`git reset --hard`、`git clean -f` は並行作業で特に危険

## パターン5: 停止時の報告フォーマット

Claude が作業を中断するとき（ユーザーの確認が必要なとき）の報告形式を指定する型です。

### 実例

```markdown
## 停止時の報告フォーマット

作業が停止境界に達したら、以下の形式で報告する:

必要アクション: [何をするか。例: PR を作成してよいか]
対象: [何に対して。例: portfolio-bridge/main]
完了した作業: [ここまでやった内容]
次のステップ: [確認後にやること]

- 「必要アクション:」は本当にユーザー操作が残っているときだけ使う
- ユーザー操作が不要なら「必要アクション: なし」と書いて作業を続ける
```

### このパターンの効果

フォーマットを決めないと、Claude の報告は冗長になりがちです。「〜をしました。〜の結果になりました。次は〜が必要ですが、〜もあるので......」と長い説明を読まないと、何を確認すればいいのか分かりません。

フォーマットを決めると、最初の1行で「何をすればいいか」が分かります。

## 5つのパターンの組み合わせ

これらのパターンは独立して使えますが、組み合わせると効果的です。

**ミニマルなプロジェクト CLAUDE.md の構成**:

```
## 外部操作の分類        ← パターン1
## ツールと接続先        ← パターン2
## 文書の優先順位        ← パターン3
```

**フルスペックの構成**（サブエージェント使用、チーム共有）:

```
## プロジェクト概要
## 外部操作の分類        ← パターン1
## ツールと接続先        ← パターン2
## 文書の優先順位        ← パターン3
## 並行作業のルール      ← パターン4
## 停止時の報告          ← パターン5
```

すべてのパターンを最初から使う必要はありません。パターン1（tier 分類）だけでも十分な効果があります。必要になったら追加してください。
