---
title: "アンチパターン集 — これをやると CLAUDE.md が壊れる"
---

ここまでの章で「良い CLAUDE.md の書き方」を扱いました。この章では逆に、「こうすると効果がなくなる」パターンを紹介します。

すべて筆者が実際にやった失敗です。

## アンチパターン1: 百科事典化

**症状**: CLAUDE.md が300行を超えている。セクションが20個以上ある。

**何が起きるか**: Claude のコンテキストウィンドウは有限です。ルールファイルが長いほど、各ルールに割ける注意力は下がります。300行あると、後半のルールは事実上無視されることがあります。

**対処**: 第10章のテクニック7で紹介した「消す基準」を定期的に適用する。また、特定ファイルにだけ関係するルールは `.claude/rules/` に分離する（第8章参照）。

**目安**: グローバルルールは150行以内、プロジェクトルールは100行以内。合計250行を超えたら分離を検討する。

## アンチパターン2: 追加だけで削除しない

**症状**: ルールが増える一方で、一度も消したことがない。

**何が起きるか**: 古いルールと新しいルールが矛盾する。「全部確認」と「push は自由」が同居して、Claude がどちらに従うか予測できなくなります。

**対処**: 月1回の棚卸しをスケジュールに入れる。具体的な棚卸しの基準は第10章のテクニック7を参照。

## アンチパターン3: 理由なしの禁止リスト

**症状**: 「〜するな」が10個並んでいるが、理由が1つも書いていない。

**何が起きるか**: Claude は律儀に禁止を守りますが、例外ケースで判断できません。例えば「git add -A を使うな」と書いただけだと、ファイルを1つ追加したいときに「git add はルール違反ですか？」と聞いてきます。理由（他のセッションの staged changes を巻き込む）と代替手段（個別に git add する）が書いてあれば、Claude は自分で判断できます。

**対処**: 第10章のテクニック1（理由を書く）とテクニック3（代替手段を添える）を参照。

## アンチパターン4: 抽象ルールの放置

**症状**: 「安全に配慮する」「品質を保つ」のような抽象ルールが書いてある。

**何が起きるか**: Claude は「安全」「品質」の定義を知りません。結果、Claude 独自の解釈で判断します。その判断がユーザーの期待と一致する保証はありません。

**対処**: 抽象ルールを具体的な条件文に分解する。「安全に配慮する」→「push は自由、PR は確認、デプロイは確認」。分解できないなら、そのルールは不要かもしれません。

## アンチパターン5: グローバルとプロジェクトの混在

**症状**: グローバル CLAUDE.md にプロジェクト固有の情報が書いてある。または、プロジェクト CLAUDE.md に個人の好みが書いてある。

**何が起きるか**:
- グローバルにプロジェクト情報 → 他のプロジェクトでノイズになる
- プロジェクトに個人の好み → チームで共有したとき全員に強制される

**対処**: 「このルール、他のプロジェクトでも使う？」と自問する。Yes ならグローバル、No ならプロジェクト。

## アンチパターン6: Memory の仕事を CLAUDE.md にやらせる

**症状**: CLAUDE.md に「現在リファクタリング中」「API v2 に移行予定」のような進行状況が書いてある。

**何が起きるか**: リファクタリングが終わっても CLAUDE.md の記述は残る。Claude は終わった作業の制約を守り続けます。

**対処**: 進行状況は Memory に書く。CLAUDE.md にはルール（恒久的な制約）だけを書く。

## アンチパターン7: 一度書いたら触らない

**症状**: CLAUDE.md を最初に書いて以来、3ヶ月間触っていない。

**何が起きるか**: プロジェクトは変化します。3ヶ月前のルールが今でも適切とは限りません。古いルールが残っていると、Claude の判断が現状とズレていきます。

**対処**: CLAUDE.md はコードと同じく「メンテナンスが必要な生きた文書」として扱う。コードを書き換えたときに、関連するルールも見直す。

## アンチパターンのチェックリスト

月1回の棚卸しで確認する項目です。

- [ ] ファイルは250行以内か？
- [ ] 最後に追加してから、何か消したか？
- [ ] 「するな」に理由が書いてあるか？
- [ ] 抽象的なまま放置されているルールはないか？
- [ ] グローバルとプロジェクトの内容が混ざっていないか？
- [ ] 進行状況メモが CLAUDE.md に混入していないか？
- [ ] 1ヶ月以上触っていないセクションはないか？

## この章のまとめ

CLAUDE.md が効かなくなる最大の原因は「肥大化」と「放置」です。

書くのと同じくらい、消すことと見直すことに力を入れてください。
