title: "次のステップ — CLAUDE.md の先にあるもの"
---

この本では CLAUDE.md の書き方・育て方・捨て方を扱いました。最後の章では、CLAUDE.md を使いこなした先に何があるかを紹介します。

## ステップ1: .claude/rules/ でルールを分離する

CLAUDE.md が100行を超えてきたら、パスベースルール（`.claude/rules/`）への分離を検討します。

- テストの規約は `rules/test-conventions.md` に
- API の設計規約は `rules/api-design.md` に
- フロントエンドのスタイル規約は `rules/frontend-style.md` に

CLAUDE.md には「全ファイルに常に適用するルール」だけを残します。

## ステップ2: Memory を活用する

Memory は CLAUDE.md の補完です。

- **ユーザー情報**: 自分の技術的なバックグラウンドを Memory に入れると、Claude の説明がカスタマイズされます
- **フィードバック**: 小さな修正指示は Memory に蓄積させて、頻繁に参照されるものだけ CLAUDE.md に昇格
- **プロジェクト状況**: 進行中のリファクタやマイグレーションの文脈を Memory に保存

## ステップ3: サブエージェントとの組み合わせ

Claude Code のサブエージェント機能を使うと、複数の作業を並列で進められます。

CLAUDE.md に並行作業のルールを書いておくと、サブエージェントが互いの作業を壊す事故を防げます。

```markdown
- shared/ ディレクトリは並列編集しない
- git add -A を使わない
- ファイルを編集する前に git status を確認する
```

## ステップ4: チームで CLAUDE.md を共有する

個人で CLAUDE.md を育てたら、チームで共有する段階が来ます。

### 共有のポイント

1. **個人の好みを分離する**: 「日本語で回答」「コミット署名」のような個人設定はグローバル CLAUDE.md に移す
2. **チーム固有のルールを追加する**: レビュー規約、ブランチ命名規則、テストカバレッジ基準
3. **onboarding ツールとして使う**: 新メンバーが最初に読む文書として CLAUDE.md を活用する

### CLAUDE.md がチームに与える効果

- 「Claude が書いたコードがチームの規約に合わない」問題が減る
- 「Claude にプロジェクトの文脈を毎回説明する」時間が減る
- チームの暗黙知が明文化される（Claude に伝わる形で書くと、人間にも伝わる）

## ステップ5: 他のツールの設定ファイルとの比較

CLAUDE.md は Claude Code のための設定ファイルですが、同じ発想は他の AI コーディングツールにも適用できます。

| ツール | 設定ファイル | 特徴 |
|--------|-----------|------|
| Claude Code | `CLAUDE.md` | Markdown、階層設計（グローバル/プロジェクト/rules/）、Memory 連携 |
| Cursor | `.cursorrules` | Markdown、プロジェクトルートに1ファイル |
| GitHub Copilot | `.github/copilot-instructions.md` | Markdown、リポジトリ単位 |

Claude Code の CLAUDE.md が他のツールと比べて特徴的なのは：

- **階層設計**: グローバル → プロジェクト → パスベース → Memory の4層
- **Memory との連携**: ルール（命令）と知識（記録）の分離
- **書き方の柔軟性**: 見出し、リスト、コードブロック、表など Markdown のフル機能が使える

この本で学んだテクニック（理由を書く、条件を具体化する、定期的に削る）は、`.cursorrules` や `copilot-instructions.md` を書くときにもそのまま使えます。

## この本で伝えたかった3つのこと

15章を通して、繰り返し伝えてきたメッセージは3つです。

1. **事故から書く**: 「あったほうがよさそう」で書いたルールは消える。「これがないと壊れる」から書いたルールだけが残る
2. **定期的に削る**: ルールは増える一方。消す基準を持たないと、ファイルが肥大化して効果がなくなる
3. **育て続ける**: CLAUDE.md は完成するものではない。プロジェクトが変われば、ルールも変わる

## おわりに

CLAUDE.md は AI との協働の「契約書」です。

何を任せるか。何を止めるか。何を覚えておくか。

最初は「とりあえず書いてみよう」で十分です。第13章のテンプレートをコピペして、1週間使ってみてください。事故が起きたらルールを追加し、不要になったら消す。そのサイクルを1ヶ月回すと、あなたのプロジェクトに最適化された CLAUDE.md が育ちます。

この本がそのきっかけになれば幸いです。
