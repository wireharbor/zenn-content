この本では CLAUDE.md の書き方・育て方・捨て方を扱いました。最後の章では、CLAUDE.md の限界と、その先にある Claude Code の機能を紹介します。

## CLAUDE.md の限界 — 対処手段の優先順位

1ヶ月の運用で気づいた最も重要なことは、**CLAUDE.md のルールは最も強制力が低い手段**だということです。

問題に対処する手段には強制力の序列があります。

| 優先順位 | 手段 | 強制力 | 例 |
|---------|------|-------|-----|
| 1 | フロー埋め込み | 最強 | コードの中に検証ロジックを入れる |
| 2 | Hook | 強い | ファイル保存後に自動で lint を走らせる |
| 3 | 関数・スクリプト | 中程度 | 手順をスクリプト化して呼ぶだけにする |
| 4 | ルール（CLAUDE.md） | 最も弱い | 「〜してください」と書く |

CLAUDE.md に「テストを必ず実行すること」と書くのは優先順位4です。コンテキストが長くなると見落としますし、Claude が「今回は不要」と判断することもあります。一方、Hook で「コミット前にテストを自動実行」とすれば優先順位2になり、Claude の判断に関係なく実行されます。

**「〜したら〜せよ」はルールではなく Hook の候補。「〜を考慮せよ」だけがルールの領域。** この区別が、1ヶ月運用してたどり着いた結論です。

## Hooks: 実際に動いている設定

私のプロジェクトでは、`rules/` ディレクトリのファイルを編集するたびに全プラットフォームへ自動配布する Hook を使っています。

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [{
          "type": "command",
          "command": "bash scripts/auto_sync_rules_hook.sh"
        }]
      }
    ]
  }
}
```

`Edit` または `Write` ツールが使われるたびに、sync スクリプトが自動実行されます。このスクリプトは `rules/` 配下のファイルが変更されたかを検出し、変更があればグローバル（`~/.claude/`）とプロジェクト（`.claude/rules/`）の両方にコピーします。

これを CLAUDE.md のルールとして「rules/ を編集したら sync スクリプトを実行してください」と書くこともできます。でもそれは優先順位4です。実際に書き忘れが何度か起きたので、Hook に移行しました。以降、同期忘れはゼロです。

**使い分けの基準:**
- 「〜を考慮せよ」→ CLAUDE.md のルール
- 「〜したら必ず〜せよ」→ Hook が適切
- 「〜を禁止する」→ Hook で検出してブロック

## カスタムスラッシュコマンド（Skills）

`.claude/commands/` ディレクトリにマークダウンファイルを置くと、`/command-name` で呼び出せるカスタムコマンドになります。

```markdown
<!-- .claude/commands/review.md -->
以下の手順でコードレビューを行ってください:
1. git diff で変更内容を確認
2. セキュリティ上の問題がないか確認
3. テストが追加されているか確認
4. 改善提案を3つ以内で出す
```

CLAUDE.md には「常に適用するルール」を、カスタムコマンドには「必要なときに呼び出す手順」を置きます。CLAUDE.md にすべてを詰め込もうとすると肥大化するので、手順の定型化にはカスタムコマンドの方が向いています。

## AGENTS.md: サブエージェント専用の指示

`AGENTS.md` は、Claude Code がサブエージェントを起動するときに読み込まれるファイルです。CLAUDE.md がメインの作業者向けなら、AGENTS.md はサブエージェント向けの指示書です。

```markdown
<!-- AGENTS.md -->
- 変更は最小限にする
- 他のファイルの変更を巻き込まない
- 完了したら結果を簡潔に報告する
```

サブエージェントは独立したコンテキストで動くため、CLAUDE.md のルールの一部しか見えません。並行作業のルール（shared ディレクトリは並列編集しない等）は AGENTS.md に書いた方が確実に届きます。

## CI/CD での CLAUDE.md 活用

CLAUDE.md はリポジトリにコミットするファイルです。つまり CI/CD パイプラインでも活用できます。

**活用例:**
- **PR レビューの自動化**: CI で Claude Code を呼び出し、CLAUDE.md のルールに基づいてコードレビューを実行
- **ルールの整合性チェック**: CLAUDE.md の行数や構造を CI でチェックし、肥大化を検知
- **チーム全体の規約統一**: CLAUDE.md をモノレポの共有パッケージとして管理し、サブプロジェクトに配布

CLAUDE.md をコードと同じくバージョン管理することで、「いつ、誰が、なぜルールを変えたか」が git log に残ります。

## 他のツールの設定ファイルとの比較

CLAUDE.md の設計思想は、他の AI コーディングツールにも応用できます。

| ツール | 設定ファイル | 階層 | 特徴 |
|--------|-----------|------|------|
| Claude Code | `CLAUDE.md`, `.claude/rules/` | グローバル → プロジェクト → パスベース → Memory | 4層設計 + Memory 連携 |
| Cursor | `.cursor/rules/` (旧: `.cursorrules`) | プロジェクト → パスベース | `.mdc` 形式、glob パターンで適用範囲を指定 |
| GitHub Copilot | `.github/copilot-instructions.md` | リポジトリ単位 | 1ファイル、Markdown |

Cursor は 2025 年後半から `.cursor/rules/` ディレクトリでのパスベースルールをサポートしており、Claude Code の `.claude/rules/` と似た設計になっています。

この本で学んだテクニック（理由を書く、条件を具体化する、定期的に削る）は、どのツールの設定ファイルを書くときにもそのまま使えます。

## この本で伝えたかった3つのこと

14章を通して、繰り返し伝えてきたメッセージは3つです。

1. **事故から書く**: 「あったほうがよさそう」で書いたルールは消える。「これがないと壊れる」から書いたルールだけが残る
2. **定期的に削る**: ルールは増える一方。消す基準を持たないと、ファイルが肥大化して効果がなくなる
3. **育て続ける**: CLAUDE.md は完成するものではない。プロジェクトが変われば、ルールも変わる

## おわりに

CLAUDE.md は AI との協働の「契約書」です。

何を任せるか。何を止めるか。何を覚えておくか。

最初は「とりあえず書いてみよう」で十分です。第13章のテンプレートをコピーして、1週間使ってみてください。事故が起きたらルールを追加し、不要になったら消す。そのサイクルを1ヶ月回すと、あなたのプロジェクトに最適化された CLAUDE.md が育ちます。
