ここから第 III 部（セキュリティ設計）に入ります。第 II 部のエラー図鑑とは違い、ここから先は **「どう決めたか」を書く章** です。正解が 1 つあるわけではなく、自分の許容リスクに応じて決める判断の話になります。

この章では、自分のポートフォリオデータを公開 URL 経由で AI アプリに返す設計のとき、**何を公開層に載せて、何を載せないか** の境界をどう決めるかを書きます。

## 筆者の結論（先に出す）

筆者は自作のポートフォリオ API で次の方針を採りました。

1. **生の個人資産データ（銘柄名、構成比、NAV の絶対額、取引履歴）は公開 URL の JSON レスポンスに載せない**
2. 公開 URL に載せるのは、個別の銘柄が特定できない **集計値・比率・感応度** だけ（例: レバレッジ倍率、ドローダウン %、為替リスクの NAV 比）
3. 生の銘柄データは、認証を通過したユーザー向けの HTML ダッシュボードでのみ表示する（JSON では返さない）
4. 認証なしで叩ける公開エンドポイントは `/health` だけにする

この方針にした理由は 3 つあります。

- 万が一 URL が漏れても、他人は自分のポートフォリオ全体像を組み立てられない
- AI アプリ（ChatGPT / Claude.ai）が答えに使うのは集計値だけで充分で、細かい銘柄は必要ない
- 細部まで返すと Claude や GPT が長大な JSON を読み込んで答えが浅くなる（実務でも結果が悪くなる）

以下は、この結論に至った判断軸と、他の選択肢を採らなかった理由の話です。

## 公開 URL に載せる前に 3 つ問いを立てる

何を載せるかを決める前に、自分の設計に対して次の 3 つを問います。

### Q1: 漏れたら誰が困るか

URL が流出したときの最悪ケースを具体化します。

- 証券会社の口座にログインできるか → No なら直接的な金銭被害は限定的
- 銘柄名と建玉がわかると信用取引の踏み上げ標的になるか → レバレッジ運用者は Yes
- 資産額がわかると社会的不都合があるか → ネット上の識別情報と紐づく可能性あり

筆者の場合、Q2 と Q3 が両方 Yes でした。銘柄と構成比は公開しない、という結論の根拠がここに集中しています。

### Q2: 攻撃者が API を悪用したら何ができるか

API 呼び出しを繰り返されたときに起きることを考えます。

- 無料枠のクラウドを使い潰され、翌月から使えなくなる（DoS 的な利用）
- Google Sheets の API クォータを使い切り、他の正当な利用がブロックされる
- レスポンスの履歴を集めて取引行動を推定される

対処は「認証を入れる」か「レート制限を掛ける」かの 2 択です。認証の詳細は次の第 8 章で扱います。

### Q3: 自分以外が読むことがあるか

AI に返すデータは最終的に ChatGPT / Claude.ai の会話ログに入ります。

- OpenAI / Anthropic は会話内容をモデル学習や運用改善に使う可能性がある（プランと設定による）
- 端末を紛失した場合、アプリのログイン状態で資産全体が見られる
- スマホを他人に貸したり、画面を覗かれたりする場面

AI の会話に銘柄構成を送ってしまえば、そのデータの管理権は自分の手を離れます。これが「認証後ですら、返すデータを絞っておく理由」です。

## 筆者のアーキテクチャ: 公開層を 3 段に切る

3 つの問いに対する筆者の答えが、次の 3 層構造です。

```
第 1 層 (Tier A): 認証なしで誰でも叩ける
  /health                — サーバーが生きているか（true/false）だけ返す
  /openapi.json          — API スキーマ（秘匿しても意味がない）

第 2 層 (Tier B): 認証を通過したユーザー向け JSON
  /portfolio/summary     — レバレッジ倍率・DD%・為替リスク NAV 比のみ
  /portfolio/stress-test — ストレスシナリオごとの NAV 増減率(%)

第 3 層 (Tier C): 認証 + ブラウザ表示用 HTML
  /dashboard             — 銘柄名・構成比・NAV 絶対額を含む HTML
                            （JSON では返さない、クライアント側表示のみ）
```

AI アプリ（ChatGPT / Claude.ai）からは Tier B だけを叩かせます。Tier C の HTML ダッシュボードは、自分がブラウザで開いて見るためのもので、AI との連携では使いません。

### なぜ Tier B の JSON に銘柄名を入れないか

AI の答えに銘柄名は要らないからです。「A 銘柄が 2% 下がったら NAV がこう動く」と個別銘柄を指定した問いを AI にさせる場面は、筆者の 3 ヶ月の運用で一度もありませんでした。ほとんどは「今の維持率」「今日の変化」「気になる点」など、集計レベルの問いです。

集計値だけで答えられる問いしかしないのに、データだけ細かく返すのは攻撃面を広げるだけです。

### なぜ Tier C を JSON ではなく HTML にしているか

JSON で返すと AI が読んでしまうからです。AI を介さず **自分だけが見たい** 情報（銘柄名付きの全体像）は、ブラウザで直接開いて見るだけにして、API 経由で AI に渡す経路を閉じています。HTML を返すエンドポイント自体は認証済みユーザーにしか見せません。

## 他の選択肢と、採らなかった理由

### 案 A: 完全公開（認証なしで全部返す）

一番シンプルだが、Q1-Q3 のどれに対しても脆弱です。個人運用ではまず採用すべきではありません。

### 案 B: ローカル専用（クラウドに出さず手元の PC でだけ動かす）

最も安全ですが、モバイルの AI アプリから自分のデータを呼べなくなります。自宅サーバー + VPN で家の外からも繋げる構成は可能ですが、家のネットワークが落ちると全機能が止まります。「外出先で維持率を確認したい」が本書の読者の主要動機なので、ローカル専用は実用から外れます。

### 案 C: 署名付きリンク（signed URL）

アクセスのたびに期限付きの署名 URL を発行する方式です。筆者の実装にも残していますが、既定ではオフにしています。理由:

- 期限が切れるたびに AI 側で再接続の手間が出る
- 署名 URL を誤って他所にペーストしてしまうと、期限内なら第三者にも開ける
- 実質的に「Bearer トークンの方が楽」

署名 URL が有効なのは「同じ URL を複数人で共有するが、期限だけ絞りたい」という共有前提の場面で、個人運用ではオーバースペックでした。

### 案 D: IP 制限（自宅・職場の IP からだけ許可）

モバイル回線や出先の Wi-Fi でアクセスできなくなるため、モバイル AI アプリから使う用途と両立しません。サーバーの管理画面には向くが、本書の用途とはズレます。

## 読者が自分で決めるための 3 問

この章のフレームを借りて、自分の設計を決めるときは次の 3 問に答えてみてください。

1. URL が漏れた場合の最悪ケースはどこまで許容できるか
2. AI に実際に使わせたい「問い」のレベルは集計か、銘柄単位か
3. 認証なしで叩ける層、認証ありで叩ける層、ブラウザ表示のみの層、をそれぞれ何にするか

Q2 が「集計だけでよい」なら筆者と同じ 3 層構造でほぼ足ります。「銘柄単位で聞きたい」なら Tier C の情報を Tier B に引き上げる設計になりますが、そのぶん Q1 のリスクが重くなります。そのトレードオフを一度自分の言葉で書いておくと、後の章の認証方式選び（第 8 章）や個人情報の開示範囲（第 9 章）の判断がブレません。

次の第 8 章は、Tier B / Tier C を守る **認証方式** の選び方に入ります。認証なし / API キー / Bearer トークン / OAuth / IP 制限 を並べて、用途ごとにどれを採るかを整理します。
