前章で「Tier B（認証後の集計値）と Tier C（認証後の HTML ダッシュボード）を守る」と決めました。この章ではその「守り方」の具体的な選択肢を並べます。

個人用の API で使われる認証方式は大きく 5 つです。それぞれの向き不向きを整理したあと、筆者が何をどう使っているかと、読者が自分で選ぶためのチェックリストを置きます。

## 5 つの方式を先に並べる

| 方式 | 概要 | 設定の重さ | 破られにくさ | 複数クライアント対応 |
| --- | --- | --- | --- | --- |
| 認証なし | 誰でも叩ける | 最軽 | 弱 | 考慮不要 |
| API キー（クエリ） | `?key=xxx` で URL に載せる | 軽 | 中（URL 漏れで終わる） | 困難 |
| Bearer トークン | `Authorization: Bearer xxx` ヘッダー | 軽 | 中〜強 | 可能だが手動配布 |
| OAuth 2.0 | 認可サーバーが短命 token を発行 | 重 | 強 | 得意 |
| IP 制限 | 送信元 IP で許可/拒否 | 軽〜中 | 中 | 固定 IP 環境のみ |

上から下に向かって「安全だが設定が重くなる」ではなく、用途との相性が大きく効きます。破られにくさだけでなく「AI アプリ側がその方式を話せるか」も重要です。

## それぞれの方式の中身

### 認証なし

URL を知っている誰もが叩けます。第 7 章で述べた通り、個人資産データを返す API でこれを選ぶ理由はありません。唯一使っていいのは `/health` のような「サーバーが生きているかだけを返す」エンドポイントです。

### API キー（クエリパラメータ方式）

`https://〇〇.onrender.com/portfolio?key=abcdef` のように URL に混ぜる形です。

- 利点: 最も実装が軽い
- 欠点: URL はブラウザの履歴、サーバーのアクセスログ、referer ヘッダーなどに残る。共有や転送で漏れやすい

AI アプリとの連携ではほぼ使いません。URL がログに残る経路が多すぎるためです。

### Bearer トークン

ヘッダーに `Authorization: Bearer xxxxxxxxxxxx` を付ける方式です。OpenAPI スキーマの `securitySchemes` に最も素直にマップされ、GPT Builder の Action 設定の既定の選択肢でもあります。

- 利点: GPT Builder が標準でサポート、URL に残らない、実装が簡単
- 欠点: トークンは「長期・無期限」で発行するので、漏れたら再発行まで被害が続く。複数ユーザーを管理しづらい

個人で自分だけが使う用途なら、Bearer 1 本で充分に成立します。漏れる経路を最小化する（GitHub に commit しない、環境変数経由でしか読まない）運用が肝心です。

### OAuth 2.0

認可サーバーが **短命の access token** を発行し、クライアントはそれを使って API を叩きます。refresh token で再発行する仕組みが規格に入っています。

- 利点: トークンが短命で漏れても被害が限定的、複数クライアント対応、スコープを細かく分けられる
- 欠点: 実装と運用が重い。認可サーバー（`/authorize`、`/token` エンドポイント）を自分で書くか、既存ライブラリを載せる必要がある

Claude.ai の Custom Connector は MCP サーバーを OAuth 経由で登録する仕様です。Claude 側から OAuth client として繋いでくるため、MCP サーバーを複数人で共有する想定があるなら OAuth を選ぶのが自然です。

### IP 制限

送信元 IP アドレスのホワイトリストで弾く方式です。固定 IP の自宅や職場、社内 VPN からだけ叩く想定なら強い守りになります。

- 利点: 漏れても「許可 IP から叩かれないと入れない」
- 欠点: モバイル回線や外出先 Wi-Fi からは入れない。本書の用途（外出先でモバイル AI から資産を聞く）とは噛み合わない

API 本体の守りというより、管理画面や内部運用ツールを閉じるときに向いた方式です。

## 筆者の選択: Bearer + OAuth の 2 階層

筆者は以下の 2 階層で運用しています。

### ChatGPT（GPT Builder）側: Static な Bearer トークン

理由:

- GPT Builder の Action 設定は Bearer 方式を標準でサポート
- OAuth クライアント機能が GPT Builder 側に弱く、自前の認可サーバーを書いても恩恵が薄い
- 個人が自分だけで使う前提なので、クライアント数は 1（または 2-3 台の自分の端末）で完結
- 長期運用に耐える「1 つの Bearer トークンを年に 1-2 回ローテーションする」程度の運用コストなら許容できる

Bearer トークンは `openssl rand -hex 32` で十分長い乱数を生成し、Render の環境変数に登録、ローカルの `.env` に同じ値を置いておくだけです。トークンがどこに置いてあるか常に把握しておけば、漏れたときの対処も速くなります。

### Claude.ai（Custom Connector）側: Stateless OAuth

理由:

- Claude.ai が MCP サーバーを繋ぐときは OAuth client として振る舞うので、サーバー側も OAuth を話す必要がある
- Claude 側が短命 token の再発行を勝手にやってくれるので、長期漏洩の心配が低い
- stateless OAuth にすると Render の再起動で認可状態がリセットされない（永続ディスクが要らない）

「stateless OAuth」は、state パラメータをサーバー側で覚える代わりに **暗号署名して投げ返す** 実装です。`PyJWT` + HMAC-SHA256 で書けます。自前で実装すると結構な行数になるので、ライブラリ（authlib 等）か、後述の fastmcp 系の OAuth ヘルパーに任せるのが現実的です。

### なぜ認証を 2 系統にしたか

1 系統（Bearer だけ、OAuth だけ）にできなかったのは、**GPT Builder と Claude.ai の要求が違う** ためです。

- GPT Builder は「Bearer が最もサポート厚い」。OAuth は可能だが UX がこなれていない
- Claude.ai の Custom Connector は OAuth を要求する。Bearer だけでは繋げない

両方のプラットフォームに繋ぎたいと決めた時点で、どちらかは譲る必要があり、結果として 2 系統持つことにしました。

片方のプラットフォームだけ使うなら、それに合わせて 1 系統で済ませるのが設計の健康を保ちます。

## 読者が選ぶためのチェックリスト

自分の用途に合う方式を選ぶとき、次の順に自問してください。

1. **使うプラットフォームは ChatGPT だけか、Claude.ai も繋ぐか** — Claude.ai を繋ぐなら OAuth が必要
2. **ユーザーは自分だけか、複数人か** — 自分だけなら Bearer で充分。複数人なら OAuth が必須ではないが推奨
3. **アクセス元の IP は固定か、モバイル含めて不定か** — 不定なら IP 制限は使えない
4. **トークン漏洩のリスクをどこまで下げたいか** — 気になるなら Bearer + 年次ローテーション、より厳しいなら OAuth

Bearer 単体で始めて、必要になったら OAuth を足す、という順は合理的です。最初から OAuth を書くと実装量で挫折します。

## 認証方式の運用で最低限やっておくこと

どの方式を選んでも、共通で決めておくべき運用があります。

- **トークン / 鍵を GitHub に commit しない**: 第 5 章の 6 項（secret commit 事故）で触れた通り、`.gitignore` を最初の commit より前に整える
- **ローテーション手順を手元にメモしておく**: トークンを再発行するときに、どの環境変数を差し替えて、どのサービス（Render / GPT Builder / Claude.ai の Connector）を更新し直せばよいかを書き残す
- **漏洩検知の通知を受ける**: GitHub の secret scanning、Google の不審アクセス通知などをオンにしておく
- **年に 1-2 回ローテーション**: 本文でも触れた通り、期間を決めて棚卸しも兼ねる

次の第 9 章は、認証の先で **どの情報を見せて、どこから隠すか** という個人情報の取り扱い境界に入ります。認証済みのユーザー（自分自身）に対してですら、銘柄名や NAV 絶対額を AI に渡すかどうかは別問題として残ります。その境界の話です。
