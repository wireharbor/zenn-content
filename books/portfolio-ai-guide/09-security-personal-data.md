認証を通ったユーザー（つまり自分自身）に対してですら、自分のポートフォリオ情報を **AI に渡すか、どこまで渡すか** は別の判断です。この章は、認証の先にある「AI に見せる情報の粒度」をどう決めるかの話になります。

AI に渡す = ChatGPT / Claude.ai のサーバーに送る ということです。会話ログに入り、プラットフォームのプランやユーザー設定によってはモデル学習に使われる可能性があります。認証で「他人」を弾くだけでなく、「自分でも送らない情報」を自分の API 側で止めておく判断が要ります。

## 結論: 集計値は渡す、個別の実データは渡さない

筆者のポートフォリオ API が AI に渡しているのは次のような項目です。

- **渡している**: レバレッジ倍率、ドローダウン %、感応度係数、為替リスクの NAV 比、維持率、評価損益率
- **渡していない**: 銘柄名、構成比、NAV の絶対額、通貨別の内訳、取引履歴、信用取引の具体的な建玉

この線引きは 3 つの軸に沿って決めています。

## 3 つの軸で線を引く

### 軸 1: 集計値か、個別データか

- **集計値** = 「全体として何倍か」「全体として何 % 動くか」
- **個別データ** = 「A 銘柄を N 株、B 銘柄を M 株」のような構成そのもの

AI に普段する問いは集計ベースです。「今の維持率は？」「世界株が 5% 下がったらどうなる？」「気になる点は？」。これらは集計値だけで答えられます。

逆に、個別銘柄を持っていることが漏れる情報は、集計値にほぼ無関係に流出リスクだけが増します。使わないのに送る理由はありません。

### 軸 2: 絶対値か、相対値・比率か

- **絶対値** = 「NAV は 1,234 万円」「今日の損益は +12.3 万円」
- **相対値 / 比率** = 「NAV は前日比 +0.8%」「最大 DD は -56.4%」

相対値と比率だけでも、自分が何を気にすべきかは十分に判断できます。「今日の損益率」で動きが大きい日は分かるし、「維持率」で強制決済までの距離は測れます。

絶対値を出すと、他の固有情報（「口座名義」「公開している Zenn Book の売上」等）と突き合わせれば個人の識別につながる恐れがあります。識別が起きない粒度まで下げてから渡します。

### 軸 3: 方法論か、具体的な実装戦略か

- **方法論** = 「維持率の閾値を 180% に置いて監視している」「感応度をストレステストで確認している」
- **実装戦略** = 「この相場では信用をこう使い、ファンド内先物ヘッジと組み合わせる」

方法論は AI と議論してよい。読者コンテンツとしても公開してよい。これ自体が流出しても他人が真似できる形ではありますが、真似されて困る性質のものではありません。

一方、具体的な実装戦略（どの銘柄でどうポジションを組んでいるか、どのタイミングで信用を増減させているか）は、商業的な価値 + 模倣リスク + 逆張りリスクの 3 つが絡みます。これは AI に渡さない、自分のノートや紙に書く、にとどめます。

## 線引きのテーブル

上の 3 軸で整理した、個人運用での渡す/渡さないの線引きです。

| 項目 | 渡す（認証後の AI 用 JSON に入れる） | 渡さない（API から返さない） |
| --- | --- | --- |
| レバレッジ倍率 | ○ 集計・比率なので OK | — |
| 最大ドローダウン % | ○ 比率 | — |
| 為替リスク NAV 比（%） | ○ 比率 | — |
| 維持率 | ○ 比率 | — |
| 評価損益率（%） | ○ 比率 | — |
| 感応度（世界株 -1% → NAV -X%） | ○ 比率 | — |
| 銘柄名 | — | × 個別データ、模倣・踏み上げリスク |
| 構成比（銘柄別の %） | — | × 合算すれば銘柄特定可能 |
| NAV の絶対額 | — | × 個人識別につながる |
| 今日の損益（絶対額） | — | × 絶対額 |
| 通貨別の内訳 | — | × 組み合わせで戦略が読める |
| 取引履歴（約定明細） | — | × 時系列で行動を推定される |
| 信用建玉の内訳 | — | × 実装戦略そのもの |

## 「AI の会話ログ」は想定より残る

API から渡したデータがどこに行くかの認識を揃えておきます。

1. ChatGPT / Claude.ai のサーバーで会話として保存される
2. プラットフォームの設定次第でモデル学習に使われる可能性がある
3. 個人のアカウント内でも会話履歴として数ヶ月から無期限で残る
4. モバイルアプリの通知、ロック解除前のプレビュー、スクリーンショット履歴に残ることがある
5. プラットフォームが第三者に買収される、プライバシーポリシーが変わる、等の将来リスク

「送らなくて済むものは最初から送らない」が設計原則です。「送るが、送り先で削除される」よりコストが低く、確実です。

## プラットフォームごとの設定確認

使うプラットフォームの設定で、最低これらは確認・変更しておきます。

- **ChatGPT（OpenAI）**: 設定 → データコントロール → 「Improve the model for everyone」を OFF（Plus 以上で選べる）。Custom GPT を保存するときは「Only me」で保存
- **Claude（Anthropic）**: 設定 → Privacy → 会話をモデル学習に使う設定の確認（プランで差あり）
- **Gemini（Google）**: 対象外だが、念のため「アクティビティとタイムライン」で記録を止める

これらはプラットフォーム側の設定なので、自分の API でどんなに情報を絞っても、プラットフォーム設定が甘ければそこで漏れます。API の設計とセットで確認しておく領域です。

## エッジケース: こういうときどうするか

### 税務や決算タイミングに絡む数字を AI に相談したい

確定申告のときに「この取引の譲渡損益がどう扱われるか」を AI に聞くことはあります。このときは **API 経由で資産データを渡さず、具体的な数値を自分で打ち込んで匿名の相談として話す** のが安全です。制度と一般論に関する問いは公開情報で答えられるので、自分のデータを見せる必要がありません。

### 相続・名義変更などの手続きで資産全体像を説明したい

これも AI にさせないカテゴリです。法的な質問は AI ではなく税理士・弁護士に問うのが本筋で、AI は前提知識を整理するツールとしてのみ使います。この用途で API を叩く必要はありません。

### 自分の配信 / 発信用に運用実績を整理したい

集計値（感応度、DD、レバレッジ倍率）は発信してよい、個別情報（銘柄、構成比、NAV）は発信しない、という境界は第 7 章 / 第 8 章の設計と一致します。「発信 OK」のデータだけを API の Tier B から取って、発信用の投稿原稿を AI に下書きさせる使い方は相性がよいです。

## 読者が自分で線を引くための 3 問

前の章と同じ形で 3 問に落とします。

1. 自分の API から AI に渡す JSON を紙に書いたとき、知らない人が同じ JSON を見てあなたの投資行動を推定できるか
2. 絶対値で渡すべき項目は本当にあるか（比率で代替できないか）
3. 渡さない項目を、自分のブラウザでだけ見る HTML ダッシュボード（第 7 章 Tier C）に移せるか

この 3 問で自分の線を決めると、第 III 部（セキュリティ設計）はおおむね閉じます。
