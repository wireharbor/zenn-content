第 II 部のエラー図鑑と第 III-IV 部の設計・実装の話で、クラウドは Render を例にして書いてきました。この章では Render 以外の選択肢と比較し、個人ポートフォリオ API に向くのはどれかを整理します。

比較の前提: 筆者は自作のサーバーを **Render で 3 ヶ月以上運用** しており、Render は実機の使用感を踏まえて書きます。Railway / fly.io / Vercel は仕様と小規模な検証経験に基づく記述で、本番運用まで走らせた実機データは Render ほど濃くありません。判断の参考として読んでください。仕様は 2026 年初頭時点。各社頻繁に更新されるので、最終的な選定時は公式ドキュメントも確認してください。

## 比較の軸を先に決める

個人用ポートフォリオ API で効く軸を 10 個挙げます。

1. **起動時間（cold start）**: スリープから起きるまでにかかる時間
2. **スリープ仕様**: 無料枠でスリープするか、するならどれくらいで
3. **無料枠と次の段階の料金**: 個人継続利用で現実的な月額
4. **GitHub 連携**: push で自動デプロイされるか、ブランチ指定の柔軟性
5. **Python / Node のランタイム指定**: バージョン固定の方法、対応の広さ
6. **環境変数の UI**: dashboard で編集できるか、量が多くても捌けるか
7. **永続ディスク**: ファイル書き込みがデプロイ間で残るか
8. **ログの見やすさ**: リアルタイム、検索、ダウンロード
9. **レイテンシ（通常時）**: スリープ解除後の応答速度
10. **デバッグしやすさ**: 再起動、シェル接続、ローカル再現

## プラットフォームごとの実体

### Render（筆者が使用中）

- **cold start**: 無料 Web Service で 30-60 秒。体感ではスマホから AI に聞いて返ってこないとき、1 回目の叩きで失敗することがほとんど
- **スリープ**: 無料枠は 15 分アクセスなしでスリープ。Starter プラン（月 7 ドル）でオフ可
- **料金**: 無料 / Starter $7 / Standard $25 / 永続ディスク 1GB で別途 $0.25
- **GitHub 連携**: main ブランチを watch して自動デプロイ。render.yaml でインフラ宣言可能、ただし dashboard 側が正本になるので混乱しやすい
- **Python**: `runtime.txt` でバージョン固定、`.python-version` は読まない
- **環境変数**: dashboard UI で直接編集、JSON のような長文字列もそのまま貼れる
- **永続ディスク**: Starter 以上で Disks 機能あり、無料枠では毎デプロイで消える
- **ログ**: dashboard でリアルタイム、過去 7 日分まで閲覧可、検索はそこそこ
- **通常レイテンシ**: 100-300ms（スリープしていなければ）
- **デバッグ**: shell 接続は有料、再デプロイは 1 ボタン、ローカル再現は docker で

筆者評価: 起動と慣れで **個人が最初に選ぶプラットフォームとして最もとっつきやすい**。無料枠のスリープさえ許容できれば、3 ヶ月運用で大きな事故はゼロ。欠点はスリープ由来の 504 が AI アプリ体験を壊すこと。

### Railway

- **cold start**: 無料枠 / 従量課金型。スリープは既定で無効で、常時起動寄り
- **料金**: 従量制、無料枠は月 500 時間相当。Hobby $5 で毎月 $5 クレジット、使い切ったら停止
- **GitHub 連携**: シームレス。PR ごとにプレビュー環境を作れる
- **Python**: `nixpacks.toml` で細かい制御、基本は自動検出
- **環境変数**: dashboard に加え、CLI からも流し込める。シークレット区分あり
- **永続ディスク**: Volume 機能、Hobby プランで使える
- **ログ**: CLI・dashboard で見やすい、フィルタあり
- **通常レイテンシ**: 100ms 前後
- **デバッグ**: CLI からリアルタイムログ、shell はアプリによる

評価: **常時起動が欲しくなったときの次のステップ** として Render からの移行先によく挙がる。従量課金は使った分だけの安心感がある一方、想定外の負荷で月末の請求が跳ねるリスクも持つ（使用量アラートは設定しておく）。

### fly.io

- **cold start**: 「停止済みマシンを起こす」モード（auto-start/auto-stop）があり、無料枠でもこれを使うと休眠から 1-3 秒で起きる
- **料金**: shared-cpu-1x の小型 VM が月 $0 ～ $3 程度（使用量による）、Hobby は実質無料枠あり
- **GitHub 連携**: CLI が中心、GitHub Actions でデプロイするパターンが多い
- **Python**: Dockerfile 前提。自動検出もあるが Dockerfile を書いたほうが安定
- **環境変数**: CLI で set、dashboard でも編集可
- **永続ディスク**: Volume あり、リージョン選択可
- **ログ**: CLI で tail、集計は Grafana 連携で細かく
- **通常レイテンシ**: 50-150ms（リージョンが近ければ速い）
- **デバッグ**: `fly ssh console` でコンテナに入れる、これが強い

評価: **cold start の速さが AI 連携に最適**。マシンを寝かせても 1-3 秒で起きるので、AI アプリのタイムアウトに引っかかりにくい。ただし CLI 中心で学習コストが少し高い。Docker を触ったことがある人向け。

### Vercel

- **cold start**: Serverless Function で約 300ms-1s の初回起動、以降はキャッシュで速い
- **料金**: Hobby は無料枠ありだが、Serverless Function の実行時間課金モデル
- **GitHub 連携**: 最も整った体験、preview deploy 自動化、コメントに URL 自動投稿
- **Python**: Serverless Function として対応、FastAPI も公式に動く。ただし cold start が出やすい
- **環境変数**: UI で管理、preview / production の切り分けあり
- **永続ディスク**: なし。外部 DB / S3 前提
- **ログ**: Function Logs、検索あり、Datadog 等の連携がしやすい
- **通常レイテンシ**: 50-300ms（エッジ配信が効く）
- **デバッグ**: ローカルで `vercel dev`、本番のデバッグはログ頼み

評価: **フロントエンドや Next.js 寄りの設計を持っているなら候補**。FastAPI も動くが、Vercel は本質的に「ステートレスな function」を走らせる設計で、長時間プロセスには向かない。MCP の stateless_http=True とは相性いいが、自前の OAuth エンドポイントなどを多く持つ設計だと手間が増える。

## 比較表

| 軸 | Render | Railway | fly.io | Vercel |
| --- | --- | --- | --- | --- |
| 無料枠のスリープ | 15 分 | 実質なし | auto-stop 選択式 | Serverless 特性 |
| スリープ復帰 | 30-60 秒 | — | 1-3 秒 | 300-1000ms |
| 最安プラン | $7/月 | $5/月〜 | $3/月〜 | $20/月 |
| GitHub 連携 | main watch | 高機能 | CLI 中心 | 最強 |
| Python | runtime.txt | nixpacks | Dockerfile | Function |
| 永続ディスク | 有料 | Volume | Volume | なし |
| CLI 操作 | 弱 | 強 | 中心 | 強 |
| FastAPI 常駐型 | ◎ | ◎ | ◎ | △ |
| MCP 常駐 | ◎ | ◎ | ◎ | △ |
| 学習曲線 | 低 | 低 | 中 | 低 |

「本書の用途に向く」という観点での筆者の評価:

1. **初めて触る・学習目的**: Render。UI が素直で事故りにくい
2. **日常運用 + コスト最適化**: Railway Hobby。スリープなし、月 $5 で足りる
3. **AI タイムアウト対策が厳しい**: fly.io。cold start が秒オーダー
4. **React / Next.js 主体のスタックに同居させたい**: Vercel

## なぜ筆者は Render から動いていないか

自作のサーバーを 3 ヶ月動かした範囲での判断です。

- **スリープは外部 ping + AI アプリ側の 2 回リトライ** でおおむね許容できた
- Render の **UI が初学者向けに優しく**、設定ミスをしにくい
- Starter プランでスリープを切っても **月 $7** で、移行コストの方が高い
- ログ調査で困った場面がなく、デバッグの手数が現状足りている

これは「Render が他より優れている」ではなく、「今の運用要件で Render 以外に乗り換える理由が積み上がっていない」という消極的な選択です。本書の用途を超えて別プロジェクトを同居させる段階で、Railway か fly.io に移る判断が出る可能性が高い想定です。

## 移行が視野に入る条件

次のどれかに当たったら、別プラットフォームへの移行を本気で検討します。

- AI アプリからの 30 秒タイムアウトで **体感を明らかに損なう** ようになった → fly.io
- 毎月の請求が $7 を超えて上昇している（データベース / 永続ディスク / 複数サービスが増えた） → Railway
- フロントエンドを React / Next.js で書き足したくなった → Vercel に寄せる
- 同じサーバーで常駐ジョブ（cron、長時間プロセス）を増やしたい → Railway / fly.io

逆に、これらのどれにも該当しないなら、Render を延命する方が合理的です。

次の第 13 章は、もし移行を決めたときに **最小コストで移す手順** のランブックです。環境変数・依存関係・GitHub 連携・ドメイン切り替えまで、順を追って書きます。
