ここから第 VI 部（データ設計）に入ります。プラットフォームを移してもデータの扱いは残ります。この章では **どのくらいの頻度で、どの仕組みでデータを更新するか** を決める話をします。

データの更新方法には大きく 3 つのアプローチがあります。

- **cron**: 時間で回す（毎日 9:00 に実行、など）
- **webhook**: 元データ側の変更通知でトリガーする
- **on-demand**: ユーザー（= AI）が叩いたタイミングで取りに行く

どれを選ぶかで、新鮮さとコストとエラー耐性のバランスが変わります。

## 個人用ポートフォリオ API で効く更新サイクル

前提を 1 つ整えます。個人運用のポートフォリオ API で読むデータのうち、**リアルタイムで動くもの** はほぼありません。

- 銘柄の現在価格: リアルタイムで動く（が、取得元が有料 API でない限り取れない）
- NAV（評価額）: 終値ベースで 1 日 1 回確定
- 維持率: 終値を使って 1 日 1 回確定
- 保有銘柄・構成比: 基本的に変わらない（約定があった日だけ動く）

つまり、個人用途の更新サイクルは **「終値が出た後に 1 回更新する」** が核で、それ以上の頻度はほぼ要りません。分単位・秒単位の更新を追いかけると、コストが跳ね上がるわりに AI への回答品質は上がらないことが多いです。

筆者の運用は以下に落ち着きました。

- **メインの更新**: 1 日 1 回、終値が出てから手動または cron で実行
- **例外**: 相場が大きく動いた日に追加で手動実行
- **AI が叩くとき**: スナップショットを読むだけ（再計算しない）

以降、3 アプローチを順に見ていきます。

## 1. cron（時間で回す）

決まった時刻にスクリプトを起動して、データ更新を走らせる方式です。

### どう書くか

- **クラウド側に cron 機能があれば使う**: Render Cron Jobs（Starter 以上）、Railway の Cron Trigger、fly.io の Machine Schedule など
- **無料枠で済ませたい**: GitHub Actions の `schedule` トリガー（cron 記法）が使える。無料で動き、コードと同じリポジトリに置ける
- **ローカル**: macOS / Linux は crontab、Windows はタスクスケジューラ

GitHub Actions は `.github/workflows/daily-snapshot.yml` に cron を書いて、当日の終値を取り、計算した JSON を同じリポジトリに commit する流れを作れます。commit したあと、Render / Railway が GitHub 連携で拾って自動デプロイすれば、クラウド側のスナップショットが最新に揃います。

### 向いている場面

- 更新タイミングが固定できる（毎日 16:30 など）
- 元データ側からの通知に依存したくない
- 自前でサーバーを管理したくない（GitHub Actions で済む）

### 注意点

- **タイムゾーン**: cron は多くが UTC で回る。JST の 16:30 に走らせたいなら UTC 7:30 と書く。タイムゾーンの記述ミスが本番で出るとそこから 24 時間気づかないこともある
- **失敗の検知**: cron が失敗したときに気付く仕組みを足しておく（GitHub Actions ならメール通知、外部なら Slack 通知など）
- **環境変数の取り扱い**: cron の実行環境が手元と違うと credentials 関連でコケやすい

## 2. webhook（元データ側の通知でトリガー）

Google Sheets の変更、証券会社からの約定通知、銀行の明細更新、といった **元データ側のイベント** でスクリプトを起動する方式です。

### どう書くか

- Google Sheets は Apps Script の `onEdit` トリガーで外部 URL に POST できる
- GitHub は push webhook で外部 URL を叩ける
- 証券会社・銀行は webhook を持たないことが多い

### 向いている場面

- データが不定期に更新される
- 更新があったタイミングで即座に最新にしたい
- 更新頻度が読めず、cron では無駄が多いか遅すぎる

### 注意点

- 通知先の URL を公開すると、**誰でも「更新されたフリ」で叩ける** ので、Webhook Secret で署名検証が必須
- 連続して更新されると webhook が嵐になり、スクリプトが同時実行される。排他制御（ロック）が要る
- 通知の信頼性（元データ側が通知を落とすことがある）を考え、cron でのバックアップ更新を同居させるケースが多い

個人用途では、webhook だけでデータを新鮮にする設計は複雑になりすぎて割に合わないことが多いです。「cron で 1 日 1 回 + 手動再実行」で充分回ることがほとんどです。

## 3. on-demand（叩かれたときに取りに行く）

AI（や人）が API を叩いたタイミングで、その場で元データを読んで返す方式です。

### どう書くか

- FastAPI のエンドポイント内で直接 `gspread.open_by_key(...)` を呼ぶ
- 取得したデータをその場で整形して返す
- キャッシュなし、毎回新鮮

### 向いている場面

- アクセス頻度が低い（1 日 1-2 回）
- データ量が少なく、取得に時間がかからない
- 常に最新を返したい

### 注意点

- **AI アプリから叩くたびに Google Sheets API を叩く** → 1 日数十回なら問題ないが、頻度が上がるとクォータに触れる
- **レスポンスが遅くなる** → Sheets 取得に 2-3 秒かかると AI アプリ側でタイムアウトしやすい
- **シート側に一時的な問題がある** → 毎回エラーを伝搬する（スナップショットがないとデータが全く返らなくなる）

## 筆者の選択: cron で日次 + スナップショットを読むだけの on-demand

筆者のサーバーはこの組み合わせです。

### 毎日 1 回の cron でスナップショットを更新

シートからデータを取り、NAV / 維持率 / 感応度などを計算して、次の 3 種類のファイルをリポジトリに commit します。

- 当日の主要指標を入れた Markdown（基準日、NAV、レバレッジ、維持率など）
- 過去の NAV 時系列を追記する JSON
- ストレスシナリオごとの NAV 変化予測（前日比）を入れた JSON

### AI が叩くときは on-demand で **スナップショットを読むだけ**

MCP tool と REST エンドポイントは、シートには触らず、上の 3 ファイルを読むだけで応答します。計算済みなので 100ms 以内に返り、タイムアウトの心配がありません。

### 取りこぼし対処

相場が大きく動いた日に追加で手動実行したいとき、`python update_snapshot.py` を叩くと cron と同じ処理が走るようにしてあります。cron と手動実行が同じコードを踏むのが肝です。

## 朝一の「データが古い」問題

cron の実行時刻と、ユーザーが API を叩く時刻のズレで、「今朝確認したけどデータが昨日のまま」という状況が出ます。

- cron が **UTC で設定されていて、JST の朝には走っていない** → タイムゾーン設定ミス
- cron が 16:00 JST に走るが、終値の確定は 15:45 JST で 17:00 以降のほうが確実 → 時刻設定の再調整
- cron は走っているが、処理の途中で失敗している → 失敗通知が無いので気付けない

対処は 3 つ。

1. **タイムゾーンを明示**: GitHub Actions なら `TZ: "Asia/Tokyo"` の環境変数と UTC 換算の cron を併記
2. **レスポンスに `as_of` タイムスタンプを必ず含める**: AI が「このデータは `2026-04-18 16:05 JST` 時点のもの」と回答で使えるようにする。古ければ AI が自発的に「今朝のデータではない」と告げてくれる
3. **失敗検知**: cron が走ったあと `as_of` が今日の日付になっているかの監視を 1 本足す

## 読者が決める 3 問

1. **ポートフォリオの更新頻度は日次で足りるか、もっと頻繁か** — 日次なら cron、それより速いなら webhook 検討
2. **元データ（Sheets / 証券会社）は変更通知を出せるか** — 出せないなら webhook は事実上使えない
3. **AI からの毎回のアクセスで元データを触る必要があるか** — ないならキャッシュ層（= スナップショット）を挟む

個人用途の 9 割は「日次 cron + スナップショット読むだけ」で充分です。凝る前にこの形で 1 ヶ月回してみて、運用で物足りなくなったら webhook や頻度を上げる順に検討するのが安全です。
