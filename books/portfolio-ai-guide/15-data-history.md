前の章で「cron で日々スナップショットを作る」ところまで来ました。この章は **作ったスナップショットをどこに、どの粒度で貯めるか** の話です。

履歴を持つと何ができるようになるかを先に置きます。

- AI に「過去 1 年の最大ドローダウンは？」と聞ける
- 「1 ヶ月前と比べて何が変わった？」に答えられる
- 運用レポートを月次・年次で自動生成できる
- 将来の自分が「あの時期にどう動いていたか」を振り返れる

つまり履歴は「自分と AI のための時系列記憶」です。これをどのくらいの深さで持つかで、運用の振り返り能力が決まります。

## 保存先の 3 択

個人用のポートフォリオ履歴を置く場所は、現実的には 3 つです。

### A. 平文ファイル（JSON / Markdown / CSV）+ Git

毎日のスナップショットを `history.json` のような 1 つの JSON にまとめて追記し、同じリポジトリに commit していく方式です。

- **書き込み**: 毎日 cron で数行追加、commit、push
- **読み込み**: ファイルを全読みして末尾 N 件を返す
- **履歴の完全な復元**: git log を辿れば任意の時点の状態に戻せる

コストはゼロ（既に GitHub を使っている前提で）。容量的にも、1 日の履歴が数 KB なら 5 年で 10MB 以下に収まります。

### B. sqlite（単一ファイルの DB）

アプリと同じリポジトリに `.db` ファイルを置いて、Python から sqlite3 で書き込む方式です。

- **書き込み**: INSERT / UPDATE、トランザクションで整合
- **読み込み**: SQL で範囲クエリが書ける（`WHERE date >= '2026-01-01'`）
- **Git 管理**: 可能だが、バイナリ差分が毎日出るので git が重くなる

平文よりクエリが書きやすく、量が増えたときにも強い。ただし Render / Railway のスリープで永続ディスクを持たない構成だと、sqlite ファイルが再起動ごとに消える問題があります。永続ディスクを有料で付けるか、別の場所（S3、Turso、fly.io の Volume）に置く工夫が要ります。

### C. 外部サービス（S3 / BigQuery / Firestore / Postgres）

データが肥大化、または複数サービスから参照したくなったら外部 DB に外出しする方式です。

- **S3 / R2**: JSON ファイルをオブジェクトストレージに置く。容量無制限、読み書き安価
- **BigQuery**: 集計・分析が強い。月数 GB のクエリなら無料枠内
- **Firestore / Turso**: マネージド DB。個人でも無料枠あり
- **Postgres (Supabase / Neon)**: リレーショナルで汎用

個人用途でここまでやるのは、**他の業務や分析と同居させたい** ような副作用がある場合です。ポートフォリオ 1 つだけなら過剰投資です。

## 筆者の選択: 平文 JSON + Git

筆者は A の方式です。理由:

- 個人 1 人が 1 ポートフォリオ運用するだけなら、履歴サイズは 5 年でも数 MB に収まる
- Git で版管理されるので、任意の時点の履歴状態に戻せる（「1 ヶ月前の計算ロジックで再集計」のような検証ができる）
- 開発・検証で手元に clone するだけで全履歴が見える
- 事故で書き戻したときに変更が明確（diff で差が見える）
- 外部サービスを増やさないぶん、障害点が少ない

不便さもあります。

- 範囲クエリは全読み → フィルタの Python コードを書く（大量データだと遅い）
- 複数プロセスが同時に書くと race condition の懸念（個人 cron 1 本なら問題なし）
- ファイルサイズが数 MB を超えて以降、ロード速度が目に見えて遅くなる

ポートフォリオ規模が個人 1 人分なら、上の不便さはほぼ顕在化しません。規模が増えて AI から数秒の応答が必要になった時点で、B または C に移る判断が出ます。

## 粒度: 日次か、もっと細かく持つか

何を毎日記録するかで、後でできる分析の深さが決まります。

### 最低限: 日次の集計値

- 基準日（日付）
- NAV（評価額）
- 維持率
- レバレッジ倍率
- 日次リターン（%）

これだけあれば、ドローダウン計算、年次リターン、月次パフォーマンスの比較ができます。容量も 1 日 100 bytes 程度で極軽量。

### 加えて持つと便利

- 為替リスクの NAV 比（日次）
- ストレスシナリオごとの NAV 変化予測（感応度 × 現在の市場変動）
- 保有銘柄の大分類ごとの構成比（「株式 50%、債券 30%」のような集約）

銘柄の個別レコードは **日次で追う必要が無い** ことが多いです。同じ銘柄を数ヶ月保有する運用なら、構成比の変化は月次で記録すれば充分で、日次で同じ銘柄情報を毎日書くのは冗長です。

### 細かすぎる記録の罠

「せっかくなら全部記録しておく」と日次で個別銘柄の現在価格まで記録し始めると、

- JSON ファイルが数 MB を超え、clone が重くなる
- Git の差分が毎日膨張する
- Claude / GPT に読ませるとトークンを食い潰す

集計値だけ毎日、詳細は月次かイベントドリブン、が個人運用の現実的な粒度です。

## ローテーション: 古いデータをどうするか

個人規模では基本 **ローテーションせずに持ち続ける** でよいです。5 年で 10MB 程度なら問題になりません。

肥大化が気になってきたら 3 段階でローテーションします。

1. **5 年以上前のデータは、月次に集約**（日次の粒度は捨て、月末値だけ残す）
2. **10 年以上前のデータは、別の archive ファイルに移す**（読む頻度が低いので通常ロードから外す）
3. **それでも膨らむなら、S3 にアーカイブを移す**（古いデータは見たいときに明示的にロードする）

筆者の現行は 5 年半（66 ヶ月）の月次 + 日次を抱えて 160KB 前後で、ローテーションなしで収まっています。

## 「過去の自分に戻れる」設計

Git commit でスナップショットを積んでいくと、任意の時点のデータに戻って再計算できます。これは検証・デバッグで強力です。

例えば「先月のドローダウン計算が間違っていたかもしれない」と思ったら、

```
git checkout <先月末のcommit>
python recalc.py
```

で当時の状態を完全に再現できます。この柔軟性は外部 DB では得にくい（スナップショット取ってない限り過去の状態に戻せない）ものです。

## 読者が決める 3 問

1. **履歴で何を AI に聞きたいか** — 過去のドローダウン、月次リターン、相場急変時の挙動、など。聞きたい項目を決めると必要な粒度が決まる
2. **データ量の想定は** — 個人 1 ポートフォリオなら A（平文 + Git）で充分。副業で複数ポートフォリオを追う、他プロジェクトと同居させる、等の場合は B か C へ
3. **復元可能性をどこまで求めるか** — 任意の時点の状態に戻して検証したいなら Git との相性がよい。「最新だけあればよい」なら DB 一択でも可

最初は A で始めて、本格的な容量問題が出てから B、さらに規模拡大で C に上げる、の順が安全です。
