第 VI 部の締めは、毎日蓄積した履歴を使って **「今日は何が変わったか」** を答えるためのデータ設計です。

個人がポートフォリオ AI に一番聞きたいのは「今の状態」と「昨日からの変化」の 2 つで、後者に答えるにはデータ側でちょっとした構造を持たせる必要があります。ここが単純なスナップショット保管と違う点です。

## なぜ差分を構造化しておくか

「今日の変化を教えて」に答える方法は 3 通り考えられます。

### 案 A: 2 日分のスナップショットを AI に渡して LLM に差分計算させる

AI に昨日と今日の JSON を両方渡して「差を説明して」と指示する形。

- 実装は一番楽（サーバー側で差分計算が要らない）
- でも AI は **数値計算が確実ではない**。例えば「昨日 NAV 1,234 万 → 今日 1,245 万」の増減率を計算させると、日によって答えが微妙にズレる
- 2 日分のデータがトークンを食い、AI のレスポンスが遅く・浅くなる
- 同じ問いに対する答えが呼ぶたびに変わる（再現性が低い）

### 案 B: サーバー側で叩かれるたびに差分計算する

AI が叩いたタイミングで、サーバーが 2 つのスナップショットを読み、差を計算して返す。

- LLM に計算させないので数値は正確
- リクエストごとに計算するため、サーバー側の処理時間が少し増える
- AI に渡すデータが「既に差分になっている」ので、AI は要約・解釈に集中できる

### 案 C: 事前に差分を計算してスナップショットに含める

日次の cron 実行時に、**前日比 / 前週比 / 前月比などの差分もその場で計算して** スナップショットに一緒に保存する。

- サーバーは読むだけ、計算は日次 cron の 1 回だけ
- AI は「差分が既に計算されたデータ」を受け取るので迷わない
- 複雑な計算（感応度 × 市場変動のストレスシナリオなど）もここで事前にやれる

筆者は **案 C** を採用しています。理由は次のとおり。

## 案 C（事前計算）に寄せる理由

- **LLM に数値計算させない** = 同じ問いに同じ答えが返るようになる。これが AI 運用の信頼性の土台
- **応答が速くなる** = 読むだけなので 100ms 以内。AI アプリ側のタイムアウトに余裕
- **複雑な計算を 1 箇所にまとめられる** = 事前計算のコードは Python で普通に書けばよく、テストも書ける
- **検証ができる** = 計算式を明示的にコードに書いてあるので、「この数字は何？」と聞かれたら式で説明できる

「AI にやらせれば設計が楽」という方向は魅力的ですが、金融のような数字の責任が重い領域では、確定値を LLM の外で計算しておくほうが安全です。

## 事前計算で入れておく差分の例

筆者のスナップショットに入れている差分データの型を挙げます。

### 前日比

- NAV の前日比（絶対額と %）
- 維持率の前日比（ポイント）
- レバレッジ倍率の前日比

これは 2 つの日次スナップショットから差を取るだけ。

### 当日の市場変動要因ごとの NAV インパクト

- 「世界株 -1% → NAV -2.5%」のような **感応度係数 × 実際の市場変動** の掛け算
- ドライバー別（世界株、米国株、米 10Y 金利、ドル円、など）に 5-10 項目

これは差分というより「今日、何が効いたか」の分解です。例:

```json
{
  "as_of": "2026-04-18",
  "drivers": [
    {
      "label": "世界株 -0.4%",
      "sensitivity": -2.5,
      "impact_pct": "+1.0%",
      "impact_yen_million": 120
    },
    {
      "label": "米10Y +3bp",
      "sensitivity": -1.9,
      "impact_pct": "-0.58%",
      "impact_yen_million": -23
    }
  ],
  "net_change_pct": "+0.42%"
}
```

AI に渡すとそのまま「今日は世界株 -0.4% が +1.0% 寄与しましたが、米 10Y の上昇で -0.58% 削られました」のような説明を組み立ててくれます。ドライバー単位で因子を分解してあるので、AI は文章化に集中できます。

### 前週比・前月比・前年比

- 同じ集計を週次・月次・年次でも用意する
- 「今月のパフォーマンスは？」の問いに答えられる
- 実装は該当期間のスナップショットをロードして差を取るだけ

### ドローダウンとリカバリの文脈

- 現在のドローダウン深度（ピークからの %）
- 過去の drawdown の発生日・深さ・回復日数
- 「前回ここまで深かった時期はいつか」が答えられる

## 差分を計算するときの注意点

### 時系列のズレに強くする

スナップショットが土日祝にスキップされる場合、「前日」は必ずしもカレンダー上の前日ではありません。前の「データがある日」を前日として扱う必要があります。

```python
def previous_business_day(history, today):
    dates = sorted(history.keys())
    idx = dates.index(today)
    return dates[idx - 1] if idx > 0 else None
```

この 1 本の関数をテストありで書いておくと、連休明けや祝日の差分計算が壊れません。

### 小さすぎる差分はノイズ

`0.01%` のような極小の差分まで AI に渡すと、AI はノイズを拾って回答を汚します。閾値（例: 0.1%）以下は「変化なし」にまとめる前処理を挟むと、AI の回答が締まります。

### 累積差分と単日差分を区別する

「今日の変化」と「今月の変化」は別のもの。同じ JSON に両方入れる場合、キー名で明確に分けます（`daily_change_pct`、`mtd_change_pct`、`ytd_change_pct` など）。

### 差分の方向を記号でなく意味語でも

`"+1.2%"` と `"sign": "up"` / `"direction": "positive"` を併記しておくと、AI が色分けしたり適切な動詞を選びやすくなります。

## AI への伝え方: 文章化のパターン

差分を構造化しても、AI が自然に答えを組み立てられるかは別問題です。伝え方のコツ:

### パターン 1: ドライバー分解 + ネット変化

```
今日の NAV は +0.42% 動きました。
内訳は、世界株 -0.4% の寄与で +1.0%、米 10Y の上昇で -0.58%、
その他のドライバーで ±0.0% です。
```

こういう説明を AI に生成させたいなら、入力データに **ドライバーの labels、それぞれの寄与、ネット合計** を分けて入れます。

### パターン 2: 過去文脈との対比

```
今日のドローダウンは -3.2% で、直近 1 年では 2 番目に深い水準です。
前回同程度に深かったのは 2026-01-15 の -4.1% で、
回復までに 18 営業日かかりました。
```

AI にこう答えさせたいなら、**履歴の中で比較可能なイベント** を近い過去から 1-2 件スナップショットに含めておきます。

### パターン 3: 気になる箇所を先に

```
維持率が前日比で -12 ポイント動いています（現在 168%）。
他の指標に大きな変化はありません。
```

AI にこう答えさせたいなら、スナップショットに **「最も大きく動いた指標」のメタ情報** を入れます（例: `biggest_mover: "maintenance_rate"`）。AI が気になる箇所を正確に拾えるようになります。

## 読者が決める 3 問

1. **何の差分に答えたいか** — 前日比、前週比、前月比、感応度分解のどこまで要るか
2. **計算をどこでやるか** — AI に任せるか、サーバー都度か、事前計算か。金融のような数字の正確さが要る用途は事前計算
3. **AI にどう伝えるか** — 生の差分データだけか、既に文章化のための構造（ドライバー、過去文脈）まで用意するか

第 VI 部はここで終わりです。ここまでで、**動かす → エラー対処 → 設計判断 → MCP の内部 → プラットフォーム → データ設計** という運用の骨組みを全部カバーしました。

次の第 VII 部（第 17-20 章）は、これまで組んだ基盤を使って具体的にどう運用しているか、4 つのユースケース（維持率・証拠金監視、ストレステスト、ニュース連動、リバランス）の話に入ります。設計論から使い方の話に、トーンが少し変わります。
