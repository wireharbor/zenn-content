第 VII 部からは、これまで組んだ基盤を **実際にどう運用しているか** の章に入ります。設計論から「日常でどう使うか」に、トーンが変わります。

最初のユースケースは維持率・証拠金の監視です。信用取引や先物・オプションなどレバレッジをかけた運用をしている人にとって、維持率は強制決済までの距離を測るメーターで、生存線そのものです。この章では閾値をどう決め、どう監視し、スマートフォンの AI アプリを挟んでどう確認するかを書きます。

レバレッジをかけない現物運用中心の読者は、本章の「維持率」を「資産 / 評価額の急変動」に読み替えてください。閾値設計と通知の考え方は共通です。

## 監視の目的を先にはっきりさせる

毎日 AI に「今の維持率は？」と聞くのが目的ではありません。**何が起きる前に気付きたいか** を決めてから監視を設計します。

目的の候補:

- 強制決済の 1-2 日前に気付いて追い証を入れる / ポジションを軽くする
- 月次の振り返りで、維持率の推移パターンを把握する
- 相場急変があった日に、口座がどのくらい影響を受けたか即時確認する

筆者は 1 と 3 が主目的で、2 は副産物です。1 と 3 が取れないと、レバレッジ運用の継続可能性が下がります。

## 閾値の 3 段階

維持率には 3 つの閾値を置いています。

| 閾値 | 水準 | 意味 | 行動 |
| --- | --- | --- | --- |
| 通常ライン | 180% 以上 | 通常運用 | 毎日の確認は軽め |
| 警戒ライン | 150-180% | 強制決済までの距離が縮んだ | ポジション縮小を検討、毎日チェック |
| 危険ライン | 130-150% | 強制決済が現実的な距離 | 即座に軽減・追い証、Slack / LINE 通知 |
| 強制決済ライン | 130% 未満（証券会社による） | 自動で決済される | ここに到達させない |

数字は証券会社や商品（日本株信用 / 先物 / FX）で異なります。自分の証券会社の「追証水準」と「強制決済水準」を一度調べて紙に書いておきます。

閾値は **絶対値で決める** のがコツです。「感覚的にヤバい」ではなく、130% ・150% ・180% のような固定値にすることで、相場で動揺したときでも機械的に判断できます。

## 監視に乗せるデータ

第 14-16 章で設計したデータ層の上に、監視用の指標を足します。

### 日次スナップショットに入れるもの

- `maintenance_rate`（現在の維持率、%）
- `maintenance_rate_pd_change`（前日比 ポイント）
- `stress_scenarios`（複数シナリオの維持率予測）
- `warning_level`（現在の閾値区分: `normal`、`caution`、`danger`）

`warning_level` を事前計算しておくと、AI に「今の区分」を直接渡せます。AI に「180% を下回ったら警戒、150% を下回ったら危険」と毎回説明する必要が無くなります。

### ストレスシナリオも一緒に持つ

「今 180% だから安全」は楽観的すぎます。保有しているポジションがレバレッジを含むなら、**世界株が X% 下がったら維持率が Y% になる** というストレスシナリオの結果も毎日計算して入れておきます。

```json
{
  "current_rate": 185,
  "stress_scenarios": [
    {"shock": "world_stock_-5pct", "projected_rate": 160, "warning_level": "caution"},
    {"shock": "world_stock_-10pct", "projected_rate": 138, "warning_level": "danger"},
    {"shock": "world_stock_-15pct", "projected_rate": 115, "warning_level": "forced_liquidation"}
  ]
}
```

現在は通常でも、-10% の相場変動で危険ラインに入るなら、「通常」だけ見て安心していると遅れます。

## アラートの仕組み

毎日自分で見に行くのは続きません。**閾値を超えたときだけ通知が来る** 仕組みを足します。

### 最小構成: cron + 条件分岐 + 通知 API

1. 日次 cron が走る（第 14 章の更新処理の末尾）
2. 計算結果の `warning_level` を判定
3. `caution` 以上なら Slack / Discord / LINE などの webhook を叩いて通知
4. `normal` なら何もしない

通知文の例:

```
[portfolio warning] 2026-04-18 16:05 JST
維持率 148% (-12pt from yesterday) — 警戒ライン到達
ストレス -10% で 115% (強制決済ライン)

https://your-service.onrender.com/dashboard
```

リンクを入れておけば、スマホから即ダッシュボードに飛べます。

### やりすぎに注意

- **毎日通知しない**: 毎日来る通知は読まれなくなる
- **同じ警戒レベルの連続通知を抑制**: 1 回 `danger` に入ったら、戻るまで再通知しない
- **逆方向（警戒 → 通常）への復帰も通知**: 「状況改善」の通知も 1 回だけ入れる

## 移動中の確認フロー

通知がなくても、朝や移動中に軽く確認したいことがあります。スマホの ChatGPT / Claude.ai から、次のパターンで問います。

### パターン 1: ワンショット確認

```
今の維持率は？
```

AI が `get_portfolio_summary` のような tool を呼んで、`maintenance_rate` と `warning_level` を取って返します。

### パターン 2: 俯瞰確認

```
今のポートフォリオで気になる点はある？
```

AI がスナップショット全体を読み、維持率・為替リスク・ドローダウンのうち最も気になるものを自発的にピックアップして答えます。「気になる点を指摘して」という曖昧な問いに答えられるのは、集計データがドライバー単位で構造化されているからです（第 16 章の事前計算）。

### パターン 3: 相場急変時の影響確認

```
今日世界株が 3% 下がったと聞いたけど、私の口座はどう動いた？
```

AI が `get_portfolio_summary` の `drivers` 配列（第 16 章）を読み、「世界株 -3% による NAV 影響は -7.5%」のような具体的な答えを返します。

### パターン 4: ストレステスト

```
ここから世界株がさらに 10% 下がったら、維持率はどうなる？
```

AI が `stress_scenarios` の該当項目を取って「-10% で維持率 138% = 危険ラインに入る」と答えます。

## 「AI に任せない」判断も設計する

維持率が危険ラインに入ったとき、**AI の判断を鵜呑みにしない** のも運用設計の一部です。

- 決済の意思決定は自分でする
- AI は「今どこに立っているか」を言語化する手伝いだけ
- 具体的な売買指示を AI に書かせない

本書の範囲では、AI は数字の確認と過去文脈の呼び出しに使うもので、意思決定の代理はしません。判断を手元に残すことで、結果に対する責任も手元に残ります。

## ダッシュボードと AI の使い分け

維持率監視について、ダッシュボード（ブラウザ）と AI（スマホアプリ）には住み分けがあります。

| 用途 | ダッシュボード | AI アプリ |
| --- | --- | --- |
| 毎日の軽い確認 | — | ○ ワンショットで終わる |
| 相場急変時の深掘り | ○ チャートも見たい | △ 説明文だけ |
| 月次の振り返り | ○ 時系列グラフ | ○ 集計値の要約 |
| 緊急時の判断 | ○ 全指標を俯瞰 | △ 補助として |
| 外出先の軽い確認 | △ 画面が小さい | ○ 声で聞ける |

AI アプリは「外出先・移動中の軽い確認」に最強、ダッシュボードは「腰を据えた分析・緊急時の俯瞰」に最強、というのが 3 ヶ月運用した感触です。

## 運用ルーチンの例

筆者の日々のルーチンを参考までに並べます。

- 朝: Slack に前日比 / 当日の大きなドライバー通知 → 気になる日だけダッシュボードを開く
- 昼: 特に何もしない
- 相場急変日: スマホで AI に「今の状況は？」。深掘りが必要ならダッシュボード
- 終値後: 通知が来なければ何もしない。来たら対応
- 月末: ダッシュボードで月次パフォーマンスを AI に要約させる

AI は「なにも起きていないときは静かにしてくれるが、聞けば即答する」存在として機能しています。この受動性を設計に組み込むと、監視の疲れが激減します。
