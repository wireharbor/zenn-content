第 VII 部の最後は、**定期リバランス** の提案に AI を使う話です。

リバランスは「自分が決めている構成比から乖離したら、元に戻すように売買する」作業です。「今の偏りはどのくらいか」「いつ戻すか」「どの順で動かすか」の 3 つを決めるため、判断材料の整理が肝心です。AI は、この整理部分の手伝いに向いています。

本書の他のユースケースと違い、リバランスは **売買の意思決定に直結** します。本章は「AI に判断させる」のではなく「AI に判断の整理を手伝わせる」話として読んでください。

## リバランスの 2 つのスタイル

個人のリバランスは、大きく 2 つのやり方があります。

### 期間型（四半期 / 年次）

「毎年 1 月に見直す」「四半期末に見直す」のように、**時期で決める** 方式です。

- 決めた時期に強制的に見直すので、忘れない
- 市場の動きと無関係にコストが発生する（売買手数料、税金）
- 期間と期間の間でどれだけ偏っても動かない

### 閾値型（〇〇が X% 以上動いたら）

「株式比率が 60% ± 10% の範囲を外れたら見直す」のように、**偏りの大きさで決める** 方式です。

- 必要なときだけ動くので無駄な取引が減る
- 閾値が厳しすぎると頻繁に売買する、緩すぎるとなかなか見直せない
- 監視の自動化が前提（手動で毎日チェックは続かない）

筆者は閾値型で運用していて、日次 cron で「閾値を外れたかどうか」を計算しています。期間型と閾値型を併用して、四半期末に必ず見直す + 普段は閾値を超えたら見直す、のパターンも合理的です。

## AI に任せる範囲と任せない範囲

リバランスで AI に任せられる仕事と、任せるべきでない仕事は明確に分けます。

### AI に任せる

- 現在の構成比を読んで、目標からの乖離を計算する
- 偏りが大きいアセットクラスを特定する
- 「どのアセットクラスをどのくらい動かせば元に戻るか」の案を出す
- 過去の同じような偏りからの戻しパターンを参照する

### AI に任せない

- 実際の売買注文
- 税務の最適化（損出し、益出しの調整）
- 含み損銘柄の売却タイミング（損切り判断）
- 外部要因（転職で収入が変わる、家族の事情など）を加味した判断
- 信用取引・レバレッジ商品の建玉調整

AI は **基本シナリオの下での数字の整理** に強いが、個別の事情や意思決定の重い判断はこちらで持ちます。

## 判断材料を事前に揃える

AI に良い提案をさせるために、スナップショットに次のデータを入れておきます。

### 現在の構成比と目標構成比

```json
{
  "as_of": "2026-04-18",
  "current_allocation": {
    "equity_japan": 0.25,
    "equity_us": 0.30,
    "equity_other_developed": 0.15,
    "equity_emerging": 0.05,
    "bond": 0.10,
    "leveraged_fund": 0.15
  },
  "target_allocation": {
    "equity_japan": 0.20,
    "equity_us": 0.30,
    "equity_other_developed": 0.15,
    "equity_emerging": 0.10,
    "bond": 0.10,
    "leveraged_fund": 0.15
  },
  "threshold_pct": 5
}
```

`threshold_pct: 5` は「目標から ±5% 以上外れたら対象」という意味。

### 計算した偏り

```json
{
  "deviations": [
    {"class": "equity_japan", "current": 0.25, "target": 0.20, "diff_pct": 5, "status": "over"},
    {"class": "equity_emerging", "current": 0.05, "target": 0.10, "diff_pct": -5, "status": "under"}
  ],
  "needs_rebalance": true
}
```

**どれが over でどれが under か、閾値を超えたか** まで事前計算しておきます。AI に計算させない方針は他の章と同じです。

### 過去のリバランス履歴

```json
{
  "last_rebalance": {
    "date": "2026-01-15",
    "actions": [
      {"class": "equity_japan", "action": "trim", "from_pct": 27, "to_pct": 20},
      {"class": "equity_emerging", "action": "add", "from_pct": 4, "to_pct": 10}
    ]
  }
}
```

直近のリバランスから何ヶ月経ったか、そのとき何を動かしたかが分かると、AI が「前回と同じ偏りパターンで、前回は A → B で戻した」のような答えを作れます。

## AI に投げるプロンプトパターン

### パターン 1: 現状確認

```
今のポートフォリオで、リバランスが必要な状況？
```

AI は `needs_rebalance` フラグと `deviations` を読み「はい / いいえ」と偏りの要約を返します。`needs_rebalance: false` なら「今は不要」で終わります。

### パターン 2: 具体案の提案

```
リバランスが必要な状況なら、
どのアセットクラスをどれくらい動かすと元に戻る？
```

AI は偏りを読んで「equity_japan を 5pt 減らし、equity_emerging を 5pt 増やす」のように具体案を返します。ここは算数レベルなので AI でも正確に答えます。

### パターン 3: 優先度の相談

```
動かしたいのは emerging を増やすところと japan を減らすところの 2 つ。
今すぐやる必要があるのはどっち？
```

AI は偏りの大きさ、最近の相場動向、前回リバランスからの期間を総合して答えます。ここは意見の性質があるので、**提案として読む** 粒度に留めます。

### パターン 4: 過去パターンの参照

```
前回 japan が偏ったときはどう戻した？
```

AI は `last_rebalance` を読んで、前回の動きを要約します。過去パターンを思い出すのは人間よりも AI が得意な作業です。

### パターン 5: 「今はやらない方がいい理由」を聞く

```
今リバランスを実行しない方がいい理由はある？
```

これは面白い問いで、AI は損益状況・相場のボラティリティ・税制要因などから「今動かない方がいい」側の根拠を並べてくれます。Yes バイアスへの対策として使えます。

## 実行は自分の手で

AI の提案が出たら、**実行は自分のブローカーで手作業** です。AI に自動で注文させる設計は組みません。

理由:

- 誤発注のリスクが人手より高い
- 売買手数料や税金の最適化は AI の外の知識が要る
- 「今日はやらない」の判断が人間の方が正確（家族の予定、体調、大きなニュースなど）
- 実行を自分でやることで、運用への自覚を残せる

AI は意思決定の補助、実行は人間、この分担を壊さないのが本章の要点です。

## リバランスを「毎日確認する」からやらない

日次でリバランス判定を走らせると、閾値周辺で毎日「やるべきかどうか」と聞かされます。気が散るし、判断疲れになります。

対策:

- **リバランス判定は日次で走るが、AI への通知は週次 or 必要時のみ**
- 閾値を外れた瞬間だけ通知、戻ったら 1 回通知、だけ残す
- 毎朝の挨拶（第 19 章のフロー）とは分離する

リバランスは腰を据えて見るテーマで、毎朝の軽い確認に混ぜるとどちらも質が落ちます。

## 読者が組むなら

1. **目標構成比** を決める（紙に書く）
2. 閾値を決める（5% か 10% が一般的）
3. 日次 cron で現在の構成比を計算、目標との diff を JSON に保存
4. MCP tool（例: `get_rebalance_status`）で JSON を返すエンドポイントを足す
5. AI への問い方の定型（パターン 1-5）を自分用にまとめる
6. 実行は必ず手動、AI は提案まで

最初は目標構成比の決定に時間がかかります。そこは本書の範囲外で、投資戦略そのものの話です。既に決めている構成比があれば、この章のフレームに流し込むだけで運用が整います。

## 第 VII 部のまとめ

第 VII 部（第 17-20 章）で、4 つのユースケース（維持率監視 / ストレステスト / ニュース連動 / リバランス）を書きました。

- **毎日の軽い確認**: 維持率 + ドライバーの俯瞰（第 17 章、第 19 章）
- **相場急変時の確認**: ストレステストとドライバー分解（第 17 章、第 18 章、第 19 章）
- **腰を据えた見直し**: リバランスの提案と実行（第 20 章）

共通する設計原則:

- 事前計算できるものは事前計算（AI に数字を計算させない）
- AI は要約と解釈と過去文脈の呼び出しに使う
- 意思決定と実行は人間が持つ
- 使用頻度に応じて通知・確認粒度を分ける

次の第 VIII 部（第 21-22 章）は、3 ヶ月動かして **「やらなくてよかった」と分かった過剰設計** と、**次の 3 ヶ月で試したい実験候補** を並べて、本書を閉じます。
