自作のポートフォリオ API を 3 ヶ月動かしたあとに振り返ると、**最初に作り込んだのに一度も使わなかった機能** や、**凝りすぎて逆に運用を難しくした設計** がいくつもありました。この章はそれを並べます。

本書の他の章が「こう組む」の話だったのに対し、この章は「こうは組まなくてよい」の話です。個人運用で同じ設計を始める読者が、最初の段階で余計なものを削れるようにまとめます。

## 過剰だった設計 1: 署名付きリンク（signed URL）

作ったもの: 専用の secret（例えば `LINK_SECRET` のような env）で HMAC-SHA256 署名した期限付きのダッシュボード URL を発行する仕組み。

使わなかった理由:

- 個人で自分用に使うなら Bearer 1 本で足りる
- 「他人に時限共有したい」場面が 3 ヶ月で 0 回
- 署名検証のロジックを読むたびに心理的コストがある
- 鍵の追加管理（Bearer とは別の secret）が増える

現在: コードは残しているが、env フラグで既定オフに落としてある。今から同じ設計を始める人は、最初から書かない方がよい。

## 過剰だった設計 2: MCP Resource / Prompt の実装

作ったもの: MCP の Tool / Resource / Prompt の 3 種類のうち、Resource と Prompt も対応。

使わなかった理由:

- 実際に AI から呼ばれるのは Tool のみ
- Resource は URI で参照する仕様だが、MCP クライアント側の対応がまちまちで、Claude.ai の表示でも Tool と大差なかった
- Prompt は自分の使い方ではほぼ使い道がなかった

現在: Resource は URI だけ残して利用コードは最小、Prompt は削除。第 10 章で書いた通り「Tool から始めて必要なら足す」が正解でした。

## 過剰だった設計 3: 複数プラットフォーム対応の先回り

作ったもの: ChatGPT / Claude.ai / Claude Desktop / Cline 等、複数の MCP クライアントで動くように分岐を大量に用意。

使わなかった理由:

- 3 ヶ月で使ったのは ChatGPT と Claude.ai のモバイルアプリの 2 つだけ
- Cline や Claude Desktop 向けの分岐は 1 度も踏まれなかった
- 分岐コードがあることで毎回「このクライアントでの挙動は？」を考える手間が増えた

現在: ChatGPT / Claude.ai の 2 本に絞り、他プラットフォーム向けの分岐は削除か最小化。YAGNI（You Aren't Gonna Need It）の原則を毎回思い出しています。

## 過剰だった設計 4: 日本語カスタム OAuth 承認 UI

作ったもの: Claude.ai が Custom Connector を登録するときに出る OAuth 承認画面を、日本語でデザインした自前 HTML テンプレート。

使わなかった理由:

- 承認画面を見るのは自分 1 人
- 自分が使うときに迷わないなら、テキストだけのデフォルト画面で充分
- テンプレート更新のたびに動作確認が必要で維持コストが高い

現在: テキストのみの最小画面に戻した。見た目の作り込みは、ユーザーが複数いる段階で考えれば十分でした。

## 過剰だった設計 5: 時系列データの「全期間 daily」ロード

作ったもの: 毎回の MCP tool 呼び出しで、過去 5 年分の日次 NAV 時系列を structuredContent に含めて返す。

使わなかった理由:

- 5 年分の daily データは 100KB オーダーで、ChatGPT の 45KB 上限を超える
- AI が聞く時系列は「直近 1-2 年」か「特定期間」のどちらかで、5 年全期間を毎回見る場面が無い
- 時系列を含めると応答が遅くなり、AI が要約に失敗する確率が上がる

現在: 通常応答には軽量な月次 + 直近 30 日だけ含め、日次全期間は `get_timeseries` という別 tool で必要時だけ返す構成に変更。読み込みが速くなり、AI の回答も締まりました。

## 過剰だった設計 6: 毎日の Slack 通知

作ったもの: 毎日終値後に「今日の維持率、ドライバー、NAV の変化」を Slack に投稿する cron。

使わなかった理由:

- 通常の日は読まない
- 毎日来る通知は「また来た」になり、逆に異常な日の通知も読み飛ばす
- 本当に気にすべきは「警戒ラインに入った日」だけ

現在: 閾値を超えた日だけ通知する設計に変更。通知が来たら必ず読む、という信頼性が戻りました。

## 過剰だった設計 7: 全銘柄の個別履歴

作ったもの: 保有銘柄ごとの日次評価額を履歴として持つ（第 15 章で触れた、細かすぎる記録の罠）。

使わなかった理由:

- 「A 銘柄の過去 3 ヶ月の評価額推移を見たい」が 0 回
- JSON が肥大化してロード時間が伸びる
- AI に渡してもトークンを食い潰すだけ

現在: 集計値（株式 / 債券 / レバレッジファンドの 3 分類）の日次と、個別銘柄は月次でだけ持つ。

## 過剰だった設計 8: 過度な tool 数の多さ

作ったもの: 「これを聞きたいならこの tool」という細かい意図ごとに tool を分け、7-8 個になっていた。

使わなかった理由:

- tool が多いと AI が「どれを呼ぶか」で迷う
- 半分以上の tool は 3 ヶ月で一度も AI から呼ばれなかった
- 自分で説明も面倒

現在: 中核の 3-4 tool（`get_summary`、`get_timeseries`、`get_news_drivers`、`render_dashboard`）に統合。AI がどれを呼ぶか迷わなくなり、応答が安定しました。

## なぜ最初は過剰になるか

自分で設計を振り返って気づいた「過剰設計が起きる構造」:

1. **想像で埋める**: 使う前に「こういう場面もあるだろう」で作り込んでしまう
2. **構造への美学**: 対称性や網羅性があるときれいに見え、切り落とすのが惜しくなる
3. **足すのは自信、削るのは勇気**: 書くのは気持ちいいが、消すのは判断が要る
4. **消さなくても動く**: 動いているコードを削るには理由が要る。惰性で残る

「最初は最小、運用で必要になったら足す」は頭でわかっていても、実装中は手が勝手に広げます。3 ヶ月運用して削ぎ落とす、は必要な工程なのかもしれません。

## 逆に「やってよかった」もの

過剰だったものの対極にある、**やっておいてよかった設計** も並べておきます。

### よかった 1: 事前計算して AI に数字を計算させない

第 16 章で書いた「LLM に計算させない」原則は、運用の信頼性を決定的に支えました。これは最初から入れておくべきでした。

### よかった 2: `as_of` タイムスタンプを全応答に含める

毎回のレスポンスに「このデータはいつ時点か」を明示する設計は、AI が古いデータで答えるリスクを消しました。小さい設計ですが効きました。

### よかった 3: stateless_http=True と git 管理の履歴

永続ディスクを持たず、履歴は Git で管理する構成は、プラットフォーム切り替えや再起動時の事故を全部防ぎました。第 10 章・第 15 章の通りです。

### よかった 4: 閾値を絶対値で決める

「感覚的にヤバい」ではなく、180% / 150% / 130% のような固定値にしたことで、相場で動揺した日にも機械的に判断できます（第 17 章）。

### よかった 5: 実行は人間、AI は整理係

全てのユースケースで AI の役割を「整理・要約・文章化」に限定し、意思決定と実行は人間が持つ、という役割分担を崩さなかったのは、長く使い続ける肝でした。

## 読者への提案

本書のフレーム全部を最初から作り込む必要はありません。むしろ、次の順に絞って始めて、運用で困った時点で足していくのが現実的です。

### 最初に作る（第 1 週）

- 日次 cron で集計値（NAV / 維持率 / レバレッジ倍率）をスナップショットに保存
- MCP tool 1 本で summary を返す
- GPT Builder または Claude.ai に登録、スマホから聞ける

### 次に足す（第 1 ヶ月）

- ドライバー別分解（第 16 章）
- 閾値を超えた日だけ Slack 通知（第 17 章）
- 感応度係数とストレステスト 3 シナリオ（第 18 章）

### 必要になったら足す（第 2-3 ヶ月以降）

- 認証層の強化（第 8 章）
- 複数プラットフォーム対応
- 履歴の細分化（第 15 章）
- リバランス提案（第 20 章）

この順で足していくと、**使わない機能を作る時間** が大幅に減ります。本書の全章は「運用で必要になったら戻ってくる辞書」として使ってもらうのが、書き手としての希望です。

次の最終章（第 22 章）は、本書を書いた時点で筆者が **まだ試していない実験候補** を並べて閉じます。
